## はじめに

Claude Codeを使い始めて、こんな経験はないだろうか。

- 最初は調子よく動いていたのに、だんだん的外れな回答が増える
- 同じ指示を何度も繰り返しているのに、さっき言ったことを忘れている
- 急に動作が重くなった

これらは全て **コンテキストウィンドウ** の問題である可能性が高い。

Anthropic公式のベストプラクティスには、こう書かれている。

> **ほとんどのベストプラクティスは、1つの制約に基づいている。Claudeのコンテキストウィンドウはすぐに埋まり、埋まるほどパフォーマンスが低下する。**

この記事では、Claude Codeを効果的に使うための「最も重要な基本」であるコンテキスト管理について解説する。

## コンテキストウィンドウとは

コンテキストウィンドウは、Claudeが「今、覚えていられる量」のこと。

```
+----------------------------------+
|      コンテキストウィンドウ       |
|  （Claudeの短期記憶）            |
+----------------------------------+
| あなたの指示                     |
| Claudeの回答                     |
| 読み込んだファイルの中身         |
| 実行したコマンドの結果           |
| エラーメッセージ                 |
| ...                              |
+----------------------------------+
         ↓ どんどん溜まる
```

会話を続けるほど、ファイルを読むほど、コマンドを実行するほど、この「箱」は埋まっていく。

---

## なぜ「埋まる」と問題なのか

コンテキストウィンドウが埋まると、3つの問題が起きる。

### 1. 精度が落ちる

箱がいっぱいになると、Claudeは「どの情報が重要か」を判断しにくくなる。

**例え話**：机の上に100枚の書類が積まれている状態で「さっき渡した1枚を探して」と言われても、見つけにくい。

### 2. 古い情報を「忘れる」

最初の方に伝えた指示が、後から入ってきた情報に埋もれてしまう。

**例え話**：会議の最初に言った注意事項が、2時間後には忘れられている。

### 3. 動作が重くなる

処理する情報量が増えるので、応答が遅くなる。

---

## 現在のコンテキスト量を確認する

Claude Codeで `/context` と入力すると、今どれくらい使っているか確認できる。

```
/context
```

表示例：
```
Context: 45,000 / 200,000 tokens (22%)
```

:::message
**目安**
- 50%以下：快適に動作
- 50-80%：そろそろ整理を検討
- 80%以上：パフォーマンス低下の可能性
:::

---

## コンテキストを管理する3つの方法

### 方法1：`/clear` で完全リセット

最もシンプルで効果的な方法。会話を全て消して、まっさらな状態に戻す。

```
/clear
```

**いつ使う？**
- 別の作業に切り替える時
- Claudeが的外れな回答を繰り返す時
- 「最初からやり直したい」と思った時

:::message alert
**重要：作業の切り替え時は必ず `/clear`**

公式ベストプラクティスでは、これを強く推奨している。

> 長いセッションで不要なコンテキストが溜まると、Claudeの注意が散漫になり、パフォーマンスが低下する可能性がある。タスク間で `/clear` を頻繁に使用して、コンテキストウィンドウを完全にリセットしよう。
:::

### 方法2：`/compact` で要約して圧縮

会話の内容を要約して、重要な部分だけ残す。

```
/compact
```

**いつ使う？**
- 今の会話の流れは残したいが、量を減らしたい時
- 自動圧縮（後述）が走る前に、自分で制御したい時

カスタム指示も可能：
```
/compact APIの変更内容だけ残して
```

### 方法3：サブエージェントに調査を委譲

「調査」をサブエージェントに任せると、メインの会話が汚れない。

```
サブエージェントを使って、認証システムの仕組みを調査して
```

サブエージェントは別のコンテキストで動き、結果だけを報告してくれる。

**なぜ有効？**
- 調査で大量のファイルを読んでも、メインのコンテキストは消費しない
- 必要な情報だけが要約されて返ってくる

詳しくは第5部「サブエージェント活用術」で解説。

## 自動圧縮（Auto Compaction）

コンテキストが上限に近づくと、Claude Codeは自動的に会話を圧縮する。

```
[Auto-compacting context...]
```

このメッセージが表示されたら、自動圧縮が走っている。

:::message
**自動圧縮で何が起きる？**
- 古い会話が要約される
- 重要なコード変更や決定事項は保持される
- ただし、細かいニュアンスは失われる可能性がある
:::

自動圧縮に任せるより、自分で `/clear` や `/compact` を使う方が、何が残るかをコントロールできる。

## 実践：コンテキスト管理のワークフロー

### パターン1：短いタスクの繰り返し

```
タスクA → /clear → タスクB → /clear → タスクC
```

タスクごとにリセットする。最もシンプルで確実。

### パターン2：長いタスクを段階的に進める

```
調査フェーズ → /compact → 計画フェーズ → /compact → 実装フェーズ
```

フェーズごとに圧縮して、重要な情報だけ引き継ぐ。

### パターン3：複雑な調査が必要な場合

```
メインの会話：計画と実装に集中
     ↓
サブエージェント：調査を担当（別のコンテキストで実行）
     ↓
結果だけメインに報告
```

調査でコンテキストを消費しない。

## よくある失敗パターン

### 失敗1：キッチンシンク・セッション

1つのセッションで、あれもこれも質問する。

```
タイマーアプリを作って
  ↓
（途中で）そういえばPythonの文法教えて
  ↓
（戻って）さっきのタイマーの続き
```

**問題**：関係ない情報でコンテキストが汚染される。

**対策**：話題が変わったら `/clear`。

### 失敗2：何度も修正を繰り返す

Claudeが間違える → 修正を指示 → また間違える → また修正...

```
これ違う、こうして
  ↓
まだ違う、こっちだって
  ↓
だから違うって！
```

**問題**：失敗した試行錯誤がコンテキストに残り、混乱を招く。

**対策**：2回修正しても直らなかったら、`/clear` してより具体的な指示で最初からやり直す。

### 失敗3：無限の調査

「〜について調べて」と広い指示を出す。

```
このコードベースについて調べて
```

**問題**：Claudeが何百ものファイルを読み、コンテキストが埋まる。

**対策**：調査範囲を絞るか、サブエージェントに委譲する。

## まとめ

| ポイント | 内容 |
|---------|------|
| 核心原則 | コンテキストが埋まると精度が落ちる |
| 確認方法 | `/context` で現在の使用量を見る |
| 基本対策 | 作業が切り替わったら `/clear` |
| 応用対策 | サブエージェントに調査を委譲 |

**覚えておくこと**：

> 2回修正しても直らなかったら、`/clear` してより良い最初のプロンプトを書く方が、修正を繰り返すより速い。

コンテキスト管理は地味だが、これができているかどうかでClaude Codeの使い心地が大きく変わる。
