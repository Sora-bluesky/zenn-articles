## この記事の対象読者

- GitHub を使い始めたばかりで「Issue（イシュー）って何？」という方
- エンジニアから「Issue 立てておいて」と言われて戸惑っている方
- 自分が出した報告で、開発チームにちゃんと伝わるか不安な方

技術的な知識は一切不要である。ブラウザの操作だけで完結するので、安心して読み進めてほしい。

---

## Issue とは何か

GitHub における **Issue（イシュー）** は、プロジェクト専用の「相談掲示板」のようなものである。

- 「ここが動かない！」→ **バグ報告**
- 「こんな機能がほしい」→ **機能リクエスト**
- 「この仕様、どうしましょう？」→ **相談・議論**
- 「来週までにこれをやる」→ **タスク管理**

こうした内容を 1 つずつ「チケット」として登録し、チーム全員で進捗を追えるようにする仕組みである。

### なぜメールやチャットではなく Issue なのか

メールやチャットで報告すると、以下のような問題が起きがちである。

| 手段 | よくある問題 |
|------|-------------|
| メール | 埋もれる。誰が対応中か分からない |
| チャット | 流れる。あとから検索しにくい |
| 口頭 | 記録が残らない。言った言わないになる |

Issue なら **1 件 1 ページで記録が残り**、担当者・進捗状況・関連するやり取りがすべて 1 か所にまとまる。「あの件どうなった？」と聞き回る必要がなくなるのが最大のメリットである。

## Issue を立てる手順（3 ステップ）

### ステップ 1：リポジトリの「Issues」タブを開く

対象のプロジェクト（リポジトリ）のトップページにアクセスし、画面上部のメニューにある **[Issues]** タブをクリックする。

> **リポジトリとは？**
> プロジェクトのファイル一式が保管されている場所のこと。「プロジェクトの本拠地」と考えればよい。

### ステップ 2：「New issue」ボタンを押す

画面右上にある緑色の **[New issue]** ボタンをクリックする。

:::message
ボタンが見当たらない場合は、以下の可能性がある。
- そのプロジェクトで Issue 機能がオフになっている
- 自分のアカウントに書き込み権限がない

→ プロジェクト管理者に確認しよう。
:::

**テンプレートが表示される場合もある。** プロジェクトによっては「Bug Report（バグ報告）」「Feature Request（機能リクエスト）」などの選択肢が出ることがある。内容に合ったものを選べば、記入項目があらかじめ用意されているので迷わず書ける。

### ステップ 3：内容を記入して送信する

入力画面が開いたら、以下の 2 つを記入する。

#### Title（タイトル）

何についての Issue か **一目でわかる名前** をつける。

| 悪い例 | 良い例 |
|--------|--------|
| エラーが出ます | ログイン画面でメールアドレス入力後にエラーが表示される |
| お願いします | トップページに「お知らせ」セクションを追加してほしい |
| 確認 | 利用規約ページのリンク切れを確認してほしい |

タイトルだけで内容が想像できると、一覧を見たときに探しやすくなる。

#### Comment（本文）

起こっている問題や、やりたいことを具体的に書く。書き方のコツは次のセクションで詳しく解説する。

記入が終わったら、**[Submit new issue]** ボタンを押せば完了である。

## 伝わる Issue の書き方──5 つのコツ

「とりあえず立てる」だけでも価値はあるが、以下のポイントを押さえると、エンジニアがすぐに動ける Issue になる。

### コツ 1：「期待する動き」と「実際の動き」を書く

これが最も重要なポイントである。

```markdown
## 期待する動き
「送信」ボタンを押すと、確認画面に遷移する

## 実際の動き
「送信」ボタンを押しても何も起きない（画面が変わらない）
```

「何がおかしいのか」を明確にするだけで、エンジニアの調査時間が大幅に短縮される。

### コツ 2：再現手順をステップで書く

「どの画面で」「何をしたら」その現象が起きたかを順番に書く。

```markdown
## 再現手順
1. トップページを開く
2. 右上の「ログイン」をクリック
3. メールアドレスとパスワードを入力
4. 「ログイン」ボタンをクリック
5. → 画面が真っ白になる（本来はダッシュボードが表示されるはず）
```

再現手順があると「自分の環境でも同じことが起きるか」をエンジニアがすぐ確認できる。

### コツ 3：スクリーンショットを貼る

言葉で説明するより、**画像 1 枚のほうが圧倒的に伝わる**。

GitHub の Issue 入力欄には、以下の方法で画像を貼り付けられる。

- 画像ファイルを **ドラッグ＆ドロップ** する
- **クリップボードからペースト**（スクリーンショットを撮った直後に `Ctrl + V`）

エラーメッセージが表示されている場合は、その画面をキャプチャして貼るだけで情報量が格段に増える。

### コツ 4：環境情報を添える

同じ操作でも、使っている環境によって結果が変わることがある。以下の情報があると助かる場面が多い。

```markdown
## 環境
- ブラウザ：Chrome 最新版
- OS：Windows 11
- スマホ／PC：PC
```

特にスマホとPCで動きが違う場合、この情報があるかないかで調査の速度がまったく変わる。

### コツ 5：1 つの Issue に 1 つの話題

「ボタンが反応しない件」と「文字が小さくて読みにくい件」は、別々の Issue として立てる。

1 つの Issue に複数の話題を混ぜると、以下の問題が起きる。

- 片方だけ解決しても Issue を閉じられない
- 担当者を分けにくい
- あとから検索しても見つけにくい

「1 Issue = 1 トピック」を意識するだけで、管理がぐっと楽になる。

## Issue 作成画面の右側メニュー（サイドバー）

作成画面の右側にはいくつかの設定項目がある。必須ではないが、チームのルールに合わせて設定するとよい。

| 項目 | 役割 | 設定例 |
|------|------|--------|
| **Assignees** | 担当者の指定 | 自分自身や、対応するエンジニア |
| **Labels** | 種類の分類 | `bug`（バグ）、`enhancement`（改善）、`question`（質問） |
| **Projects** | プロジェクトボードへの紐付け | カンバン形式（ToDo → Doing → Done）で進捗管理 |
| **Milestone** | 期限やバージョンの紐付け | 「v2.0 リリース」「3月末までに対応」など |

:::message
最初は **何も設定しなくても大丈夫** である。タイトルと本文さえ書けば、あとからチームメンバーが適切に設定してくれることが多い。
:::

## Issue を立てたあとの流れ

Issue は「立てて終わり」ではない。その後の基本的な流れを知っておくと、やり取りがスムーズになる。

### 1. コメントでのやり取り

Issue の下部にコメント欄がある。エンジニアから「もう少し詳しく教えてください」と質問が来ることがあるので、気づいたら返信しよう。メールの返信と同じ感覚である。

### 2. ラベルやステータスの変化

担当者が決まったり、対応中になったりすると、ラベルや Assignees が更新される。自分が立てた Issue の進捗は、Issue ページを見ればいつでも確認できる。

### 3. Issue のクローズ（完了）

問題が解決したり、対応が完了すると、Issue は **Close（クローズ）** される。自分で閉じることもできるし、エンジニアが修正コードを反映した際に自動で閉じられることもある。

## よく使う Markdown の書き方

Issue の本文では **Markdown（マークダウン）** という記法が使える。覚えておくと便利なものを厳選して紹介する。

### 見出し

```markdown
## 大見出し
### 小見出し
```

### 箇条書き

```markdown
- 項目 1
- 項目 2
  - サブ項目
```

### チェックリスト（タスクリスト）

```markdown
- [ ] 未完了のタスク
- [x] 完了したタスク
```

チェックリストは Issue の一覧画面にも進捗バーとして表示されるため、タスク管理に便利である。

### 太字・引用

```markdown
**太字にしたいテキスト**

> 引用文（他の人の発言や、参照元の文章を示す）
```

### コードやエラーメッセージ

エラーメッセージを貼り付けるときは、バッククォート 3 つで囲むと読みやすくなる。

````markdown
```
ここにエラーメッセージを貼り付ける
```
````

:::message
Markdown を覚えなくても、入力欄の上にあるツールバーのボタンで太字や箇条書きを適用できる。まずはツールバーから試してみるとよい。
:::

## テンプレート：コピーして使える Issue フォーマット

以下をコピーして Issue の本文に貼り付ければ、すぐに使える。

```markdown
## 概要
（何が起きているか、何をしてほしいかを 1〜2 行で）

## 期待する動き
（本来どうなるべきか）

## 実際の動き
（現在どうなっているか）

## 再現手順
1.
2.
3.

## 環境
- ブラウザ：
- OS：
- デバイス：PC ／ スマホ

## スクリーンショット
（あれば画像を貼り付け）
```

## やりがちな失敗と対処法

| やりがちなこと | なぜ困るか | こうすればよい |
|---------------|-----------|---------------|
| タイトルが曖昧（「エラーです」） | 一覧で内容が判別できない | 「〇〇画面で△△するとエラーになる」と具体的に |
| 本文が空、タイトルのみ | 何をすればいいか分からない | 最低限「期待する動き」と「実際の動き」を書く |
| 1 つの Issue に複数の話題 | 管理が煩雑になる | 話題ごとに Issue を分ける |
| チャットで報告して Issue を立てない | 記録が流れて消える | チャットで話した内容も Issue に転記する |
| 解決したのに閉じない | 未対応の Issue と区別がつかない | 解決を確認したら Close する |

---

## まとめ

Issue を立てることは、プロジェクトをより良くするための第一歩である。

押さえるべきポイントを振り返ろう。

1. **タイトルは具体的に** ── 一覧で見て内容が分かるように
2. **「期待」と「実際」を書く** ── 何がおかしいか明確にする
3. **再現手順をステップで** ── エンジニアがすぐ確認できるように
4. **スクリーンショットを活用** ── 画像 1 枚は文章 100 行に勝る
5. **1 Issue = 1 トピック** ── 管理しやすく、検索しやすく

「こんなこと書いていいのかな？」と迷う必要はない。完璧な報告より、**まず立てること**のほうがずっと大事である。相談ベースで気軽に立ててみよう。
