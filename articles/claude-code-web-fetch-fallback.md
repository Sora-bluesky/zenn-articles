## はじめに

2026年2月5日、Anthropicが最新モデル[Claude Opus 4.6](https://www.anthropic.com/news/claude-opus-4-6)をリリースしました。推論の再検討、長期セッションの一貫性維持、適応的思考。AIが「道具」から「一緒に考えるパートナー」に近づくための改善が詰め込まれています。

この記事は、リリース当日にOpus 4.6とClaude Code（AnthropicのAIコーディングツール）で対話しながら、Webページ取得のフォールバック機構を実装した記録です。

実装だけの話では終わりませんでした。こちらの思い込みをClaude Codeが正してきたり、逆にこちらがClaude Codeの矛盾を見つけたり。そのやりとりを、エンジニアでない方にも伝わるように書きました。

なぜWebページの取得方法にこだわるのか。毎日Xのタイムラインで二次情報に振り回されて、肝心の公式発表を見逃すことに疲れていたからです。一次情報を確実に取りに行ける仕組みが欲しかった。フォールバック機構はその第一歩でした。

## 「がっかりパターン」の正体

AIにWebページを読ませると、がっかりする場面があります。

### 人間なら簡単

```
Googleで検索 → リンクをクリック → ページが開く → 読む
```

ページが開かなければ、別のリンクを試せばいい。

### Claude Codeだとこうなる

```
私「このサイトを読んで」
  → Claude CodeがWebFetchツールを使う
    → サイトがJavaScriptで動いている
      → 空っぽのHTMLしか取得できない
        → 「取得できませんでした」
          → 私「えっ……」
```

これが「がっかりパターン」です。

実際に https://antigravity.google/ で試したところ、Claude Codeの標準ツール（WebFetch）はフォントの定義ファイルとアクセス解析のコードだけを返してきました。ページの本文は一切なし。

## 「ダメだったら別の方法で」を自動化する

人間なら「このブラウザで開けないなら、別ので試そう」と自然にやります。Claude Codeにも同じことをさせればいい。

### フォールバック（代替手段への自動切り替え）

```
1. Jina Reader（軽量ツール）で取得を試みる
  → きちんと読めた → 完了
  → 取得失敗、または中身が壊れている ↓

2. Playwright（ブラウザ自動操作）で取得する
  → 実際にブラウザを開いて、人間と同じようにページを読む
```

第一候補がダメなら、第二候補に自動で切り替わる。これがフォールバックの考え方です。

実際にantigravity.google/で試したところ、Jina Readerはコンテンツ自体は返してきました。ただし、このサイトはCSSアニメーションで文字を1文字ずつ表示する作りになっているため、返ってきた中身は `E x p e r i e n c e l i f t o f f` のように1文字ずつバラバラの1,629行。サブエージェントはこれを見て「まともに読めない」と判断し、Playwrightに切り替えました。エラーが返ってくるのではなく、中身を見て判断している点が面白い。

[Jina Reader](https://jina.ai/)はWebページの内容をテキストとして高速に取得する軽量ツール。[Playwright](https://github.com/microsoft/playwright-mcp)はMicrosoftが開発したブラウザ自動操作ツールで、JavaScriptで描画されるサイトでも人間と同じようにブラウザで開いて読めます。どちらも[MCP（Model Context Protocol）](https://modelcontextprotocol.io/)という標準規格でClaude Codeに接続しています。

## 「秘書に任せる」仕組み

フォールバックを実装するだけなら、Claude Codeに「Jinaで失敗したらPlaywrightを使え」と指示すれば済みます。ただ、それだとClaude Codeの「脳の容量」を食ってしまう。

### 作業机の問題

AIにはコンテキストウィンドウという「一度に覚えていられる量」の上限があります。作業机の広さだと思ってください。

Webページを読むと、そのページの全文が机の上に広がる。10サイト調べれば、机は資料だらけ。やがて机が一杯になって、最初の方の会話内容を忘れてしまいます。

### サブエージェントに丸投げする

Claude Codeには[サブエージェント](https://docs.anthropic.com/en/docs/claude-code/sub-agents)（[日本語](https://docs.anthropic.com/ja/docs/claude-code/sub-agents)）という仕組みがあります。専門の秘書に仕事を丸投げするようなものです。

```
【従来】社長（メインAI）が検索も取得も全部一人で処理する
  社長の机 ← 検索結果、ページ全文、エラーメッセージ……山積み

【改善後】社長が秘書に「調べてきて」と指示
  秘書の机 ← 検索結果、ページ全文、エラーメッセージ（秘書が処理）
  社長の机 ← 秘書からの報告書1枚だけ
```

秘書（サブエージェント）が自分の机で作業して、社長（メインAI）には要約だけ渡す。社長の机はきれいなままです。

### 実測した効果

10サイトを調査した場合の机の使用量を比較しました。

| やり方 | 机の使用量 | 情報取得 |
|--------|-----------|---------|
| 従来（メインAIが全部処理） | 約12,700トークン | 7/10サイト（3サイト失敗） |
| 秘書方式（サブエージェント） | 約1,080トークン | 10/10サイト（失敗時は自動代替） |

机の使用量は約92%減。しかも全サイトの情報を取得できています。

## コスト意識が設計に入っている

Claude Codeの組み込みサブエージェントには、もう一つ仕掛けがあります。

AIにはグレード（性能レベル）があって、それぞれ料金が違います。

| グレード | 得意なこと | コスト感 |
|---------|-----------|---------|
| Opus（最上位） | 深い思考、複雑な判断 | 高い |
| Sonnet（中位） | バランスの良い作業 | 中くらい |
| Haiku（軽量） | 単純な検索・取得 | 安い |

Webページを取得するだけなら、高いAIを使う必要はありません。Claude Codeは検索・取得作業をHaiku（安いAI）に任せて、考える作業はOpus（高いAI）が担当する。

会社で言えば、調査はインターンに任せて、分析と判断はベテランが行う。この分業はClaude Codeに最初から設計として組み込まれています。

## 「調べて」と言われたら必ず検索するのか

「Google Antigravityとは？」と聞いたとき、Claude Codeが必ずWeb検索するわけではありません。

```
私「Google Antigravityとは？」
  │
  Claude Codeの判断
  │
  ├── 「知ってる」→ 自分の知識だけで回答（検索しない）
  │
  └── 「最新情報が必要」→ 検索ツールを使う
```

Claude Codeには学習時に獲得した膨大な知識（2025年5月時点まで）があります。多くの質問にはそれだけで答えられる。検索ツールの出番は「知識では足りない」と判断したときだけです。

つまり、フォールバック機構が活躍するのは、Claude Codeが「自分の知識では足りない」と判断した場面に限られます。

## 実装してぶつかった「限界」

ここまでは順調でした。ただ、実装してみて初めてわかった設計上の限界があります。

### 組み込みサブエージェントは指示を無視する

Claude Codeには、ユーザーが作るカスタムサブエージェントのほかに、最初から組み込まれているビルトインサブエージェントがあります。

```
社長（メインAI）
  ├── 自分で雇った秘書（カスタム）→ 社長ルールに従う ✓
  └── 外注の調査会社（ビルトイン）→ 自社のやり方で動く ✗
```

外注先に「うちの社内システムを使ってくれ」と言っても通らないのと同じです。組み込みの調査担当（claude-code-guide）は独自のツール（ネイティブのWebFetchとWebSearchだけ）で動作し、Jina ReaderにもPlaywrightにもアクセスできない。こちらが設定したルールも読みません。

実際、Opus 4.6のリリース情報を調査させたとき、この問題が起きました。Claude Codeが組み込みの調査担当に作業を回し、その調査担当がルールを無視してネイティブのWebSearchとWebFetchを使っていた。

### なぜ起きるのか

| 要因 | 説明 |
|------|------|
| ビルトインのツールは固定 | ユーザーが変更できない |
| CLAUDE.md（設定ファイル）はビルトインに適用されない | カスタムにのみ有効 |
| サブエージェントは入れ子にできない | 秘書が別の秘書に頼むことは不可 |

### 実害は？

実害は限定的です。

- 組み込みサブエージェントはHaiku（軽量AI）で動く。コスト影響は小さい
- 独自のコンテキストで実行される。社長の机（メイン）は汚れない
- 問題はコストではなく取得成功率。WebFetchで読めないサイトがあると、そこだけ情報が欠ける

完璧ではありません。ただ、ユーザーが直接「調べて」と指示する場面ではカスタムサブエージェントが使われるので、大半のケースでフォールバックは機能します。

## 対話を通じて納得できる形にたどり着いた

今回の実装で一番の収穫は、技術そのものではなく対話のプロセスでした。

### 最初のリクエスト

> 「Playwrightを組み込んだWeb Fetchフォールバック機構を実装して」

もしClaude Codeがこの指示をそのまま実行していたら、単純な指示文を設定ファイルに書いて終わりだったでしょう。

### 実際はこうなった

```
1. 「仕組みを調査して」     → まず現状を正確に把握
2. 「実装して」             → 優先順位と方式を私に確認してきた
3. 「handoffを作成して」    → 公式ドキュメントでベストプラクティスを確認
4. 「コスト比較して」       → 実測テストで効果を検証
5. 前提が違うのでは？       → Claude Code側から私の思い込みを指摘
6. ルール無視してない？     → 私がClaude Codeの矛盾を発見
```

5番目と6番目が特に面白かった。

5番目では、私が「WebSearch→Fetchがデフォルトの流れですか？」と聞いたとき、Claude Codeは「いいえ、まず自分の知識で回答を試みます」と訂正してきました。

6番目では逆に、私がClaude Codeの行動の矛盾（ルールを設定したのにネイティブツールを使っている）を見つけて、設計上の限界が明らかになりました。

Claude Codeが私の前提を正し、私がClaude Codeの行動を正す。この双方向のフィードバックが、より良い設計にたどり着くための鍵でした。

## Opus 4.6で変わったフィードバックの質

今回のやりとりは[Claude Opus 4.6](https://www.anthropic.com/news/claude-opus-4-6)（2026年2月5日リリース）で行いました。セッションの随所に、4.6の改善が出ています。

### 前モデル（4.5）との違い

| 能力 | Opus 4.5 | Opus 4.6 |
|------|---------|---------|
| 推論の再検討 | 一度出した結論を維持しがち | 新情報に基づいて自ら結論を修正 |
| 長いセッションの一貫性 | 長くなると文脈を見失うことがあった | 長時間の対話でも一貫性を維持 |
| 計画の深さ | 基本的な計画を立てる | より慎重に計画し、前提条件を検証 |
| 自己修正 | エラーを見落とすことがあった | 自分のミスを検出・修正する能力が向上 |
| 適応的思考 | 常に同じ深さで思考 | 問題の複雑さに応じて思考の深さを自動調整 |

### 今回のセッションで見えた場面

前提への反論として、私が「Playwrightでなければ処理できない」と思い込んでいたところ、Claude Codeが実際にテストして「Jina Readerでも取得できます」と事実で訂正してきました。4.5でもできた可能性はありますが、実測データを添えて穏やかに反論してくる精度は上がっています。

長期セッションの一貫性も印象的でした。このセッションは調査→実装→検証→分析→記事作成と多段階にわたりましたが、途中で文脈を見失うことなく、前半の議論を後半の分析に正確に反映していました。

適応的思考については、こんな場面がありました。建築士が確認申請前に構造・法規・設備の各観点から図面を見直すように、私もClaude Codeに「一度立ち止まって、自分の提案を複数の視点で見直してほしい」と指示しました。するとClaude Codeはチェックリスト的な確認で済ませず、「サブエージェントの配置場所は本当にグローバルでいいのか」「CLAUDE.mdと重複しないか」といった問いを自ら立てて、回答の質を上げていた。4.6の「問題の複雑さに応じて思考の深さを自動調整する」機能が効いていると感じます。

### ただし、過信はしない

4.5でも同じ振る舞いをする可能性はあります。4.6で確実に変わったのは、これらの能力の安定性と精度です。

そして実際に、Opus 4.6でもルールを破る場面がありました（ビルトインサブエージェントがWebFetchを使った件）。AIへの過信は禁物で、人間の目による検証は今も欠かせません。

## この仕組みで何が変わったか

フォールバック機構は、それ自体がゴールではありませんでした。

やりたかったのは、Anthropic・OpenAI・Googleといった公式ソースから毎日の更新情報を自動で取得して、日本語に変換して、Obsidianに集約すること。英語の一次情報だけを読みに行き、差分だけ同期するから重複もない。大手ベンダー以外のソースも今、増やしているところです。

いま朝Obsidianを開くと、昨日のAI業界の動きが日本語で整理されて並んでいます。Xを巡回する時間がゼロになりました。

実際、フォールバック機構がない状態では公式ソースの半分近くがエラーで取得できませんでした。JavaScriptで描画されるサイト、API経由でしかコンテンツを返さないサイト。なんとかしたくてフォールバック機構を組んだ結果、Jina Reader→Playwrightの自動切り替えで取りこぼしがなくなった。

JinaとPlaywrightが最善かどうかはわかりません。有料ツールも含めれば、もっと精度の高い組み合わせがあるかもしれない。今回の仕組みはあくまで出発点で、自分の用途に合わせてツールを入れ替えていくものだと思っています。

この情報収集の仕組みの作り方は、別の記事で書くつもりです。

## まとめ

- AIのWeb取得は万能ではない。だからフォールバックが要る
- サブエージェント（秘書方式）で「脳の容量」問題を約92%削減できた
- 軽い仕事は軽いAIに。Haiku・Sonnet・Opusの使い分けでコストを抑える
- 完璧ではない。組み込みサブエージェントにはルールが届かない
- Claude Codeに言われたとおりやらせるのではなく、対話を通じて最適解にたどり着く
- Opus 4.6は「前提を疑う力」と「長期的な一貫性」が安定した。ただし過信はしない
