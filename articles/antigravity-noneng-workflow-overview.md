でも、途中で何度もやり直すことになった。

...30分経った。フォルダ名すら決まらない。

なんかセンスのいい名前にしたいけど、命名センスがない。

---

「レシート撮影できるようにして」と言ったら、スマホアプリを作り始めた。

違う違う、そうじゃない。GAS（Google Apps Script: Googleの無料プログラミング環境）でいいのに...

## 前提条件

- Google Antigravity インストール済み
  - まだの場合 → [Antigravityインストールガイド](antigravity-windows-install-guide)
- プログラミング経験は不要

## Antigravityの「スキル」とは

Antigravityには3つのカスタマイズ方法がある。本シリーズでは主に「スキル」と「ワークフロー」を使う。

| 種類 | 役割 | いつ動くか |
|------|------|-----------|
| Rules | 常に守るべきルール（コードスタイルなど） | 自動（常時） |
| Workflows | 手動で呼び出す手順（`/team`など） | 手動 |
| **Skills** | 専門知識パッケージ | **AIが自動で選んで適用** |

**スキルが重要な理由**: 「経費精算を楽にしたい」と言うだけで、AIが「これは要件定義スキルを使うべきだ」と判断して自動適用してくれる。

### Claude Code との比較

Claude Codeを使っている人向けに、概念の対応関係を整理する。

| 機能 | Antigravity | Claude Code |
|------|-------------|-------------|
| 常時適用ルール | Rules | CLAUDE.md |
| 手動呼び出し手順 | Workflows | Slash Commands |
| 専門知識の拡張 | **Skills** | **Subagents** |

**UIの違い**:

| 項目 | Antigravity | Claude Code |
|------|-------------|-------------|
| インターフェース | GUI（コックピット型） | ターミナル（コマンドライン） |
| 起動画面 | Agent Manager（管制塔） | 黒い画面にテキスト |
| 進捗確認 | 視覚的に一覧表示 | テキストで確認 |
| 作業の証拠 | スクリーンショット・動画 | ログ・差分 |

**Antigravityの「コックピット感」**:

Antigravityを起動すると、ファイル一覧ではなく「Agent Manager」という管制塔のような画面が表示される。

- **全体が見渡せる**: 複数のタスクの進捗を一覧で確認できる
- **視覚的な証拠**: AIが作成した計画書、スクリーンショット、動画を確認できる
- **委任レベルを選べる**: 「オートパイロット（全自動）」「許可制（毎回確認）」「アシスト（推奨）」から選択

飛行機のコックピットのように、「今どこにいて、何が起きているか」が一目でわかる。

**Claude Codeの「ターミナル」**:

Claude Codeは黒い画面にテキストを打ち込むスタイル。エンジニアには馴染み深いが、非エンジニアには敷居が高い。

- **高速で柔軟**: 慣れれば最速で操作できる
- **並列実行が強力**: 複数のサブエージェントを同時に走らせられる
- **独自コンテキスト**: 各サブエージェントが独立した記憶領域を持つ

:::message
**どちらを選ぶべきか？**
- **非エンジニア・視覚的に確認したい** → Antigravity がおすすめ。管制塔のようなUIで全体が見渡せる。
- **エンジニア・ターミナルに慣れている** → Claude Code。高速で柔軟な操作が可能。

本シリーズは非エンジニア向けなので、Antigravityを使用する。
:::

:::message
**公式ドキュメント**
- [Antigravity Skills](https://antigravity.google/docs/skills)
- [Claude Code Subagents](https://docs.anthropic.com/en/docs/agents-and-tools/claude-code/sub-agents)

```
ユーザー: /team

AI: 🚀 AIチーム開発ワークフロー開始

Phase 1: 要件定義 ← 今ここ
Phase 2: 設計
Phase 3: 実装
Phase 4: レビュー
Phase 5: ドキュメント作成
Phase 6: 振り返り

何を作りますか？

ユーザー: 経費精算を楽にしたい

AI: いくつか質問させてください。
    誰が使いますか？

（AIが質問しながら進めてくれる）

AI: フォルダ名は `team-expense-assistant` でいかがですか？
    （理由: チーム利用であること、業務を補助する役割を明確にするため）

ユーザー: はい

AI: ✅ フォルダを作成しました

...

AI: 🎉 プロジェクト完了！

作成されたもの:
✅ team-expense-assistant/（フォルダ）
✅ docs/DESIGN.md（設計書）
✅ src/main.gs（コード）
✅ docs/REVIEW.md（レビュー結果）
✅ README.md（使い方）
✅ docs/PRESENTATION.md（上司報告用）
✅ docs/RETROSPECTIVE.md（振り返り）
```

**ポイント**:
- `/team` と入力するだけで開始
- **AIが質問してくれるから、何を伝えればいいか迷わない**
- **フォルダ名もAIが提案してくれる（命名センス不要）**
- 各フェーズ完了後に確認があるので、スキップも可能
- 「ここで終わり」と言えばいつでも終了

## なぜこの仕組みを作ったのか

非エンジニアの個人開発で一番つらかったのは、 **「AIに何を伝えればいいかわからない」** ことだった。

| やりたいこと | 伝えたこと | AIの解釈 | 結果 |
|-------------|-----------|---------|------|
| フォルダ作成 | （30分悩む） | — | 命名で止まる |
| スマホで撮影 | 「レシート撮影」 | ネイティブアプリ開発 | 違う... |
| 転記の自動化 | 「自動入力」 | OCR精度向上 | そうじゃない... |
| 集計表が欲しい | 「月次レポート」 | BI連携 | 大げさすぎる... |

**問題は自分の伝え方だった。**

でも、何をどう伝えればいいか、非エンジニアにはわからない。

だから、 **AIの方から質問してもらう仕組み** を作った。

## 6つのフェーズ（+発展編）

| Phase | 名前 | 解決する悩み | 詳細記事 |
|-------|------|-------------|----------|
| 0 | 過去の学びを確認 | 振り返りを次に活かせない（発展編） | [詳細](antigravity-insights-skill) |
| 1 | 要件定義 | 何を伝えればいいかわからない | [詳細](antigravity-requirements-skill) |
| 2 | 設計 | どのツールを使えばいいかわからない | [詳細](antigravity-design-skill) |
| 3 | 実装 | （AIが自動で実装） | — |
| 4 | レビュー | これで合ってるか不安 | [詳細](antigravity-review-skill) |
| 5 | ドキュメント | 上司への報告資料が必要 | [詳細](antigravity-documentation-skill) |
| 6 | 振り返り | 同じ失敗を繰り返したくない | [詳細](antigravity-retrospective-skill) |

:::message
Phase 0 は発展編。基本編（Phase 1〜6）を完了してから導入する。
:::

各フェーズは「5つの質問」で進む。**複雑な指示は不要。質問に答えるだけ。**

## なぜワークフローが必要なのか

最初は自然言語で指示していた。

「経費精算ツール作って」
「レシート撮影できるようにして」
「自動入力もお願い」

これだと、 **毎回AIの解釈が違う**。

| 方式 | 結果 | 理由 |
|------|------|------|
| 自然言語で都度指示 | ブレる | AIが都度判断 |
| ワークフロー | 一貫する | 手順が決まっている |

**ワークフローを使えば、毎回同じ流れで進む。**

「今日はスマホアプリ、明日はGAS」みたいなブレがなくなる。

:::message alert
**注意**: `/team` はAntigravityのデフォルト機能ではない。
本記事の手順で自分で作成する必要がある。
一度作成すれば、フォルダごとコピーして他のプロジェクトでも再利用できる。
:::

:::message
**ワークフロー名は自由に変更できる**

本記事では `/team` という名前を使っているが、これは一例。
- `/dev` — 開発っぽい名前が好みなら
- `/start` — 「始める」という意味で
- `/project` — プロジェクト開始っぽく

ファイル名を変えるだけ: `team.md` → `dev.md` で `/dev` になる。
:::

## 全体構成

```
projects/
├── .agent/
│   └── workflows/
│       └── team.md   → /team （一気通貫開発）
├── .insights/                   ← 学びの集約（発展編）
│   └── retrospectives/
├── skills/
│   ├── requirements-gathering/  ← 要件定義スキル
│   ├── design/                  ← 設計スキル
│   ├── review/                  ← レビュースキル
│   ├── documentation/           ← ドキュメントスキル
│   ├── retrospective/           ← 振り返りスキル
│   └── insights/                ← 学びの継承スキル（発展編）
└── [プロジェクトフォルダ]/       ← 開発時に自動生成
    ├── docs/
    │   ├── DESIGN.md
    │   ├── REVIEW.md
    │   ├── PRESENTATION.md
    │   └── RETROSPECTIVE.md
    ├── src/
    └── README.md
```

## セットアップ手順

### Step 1: フォルダ構成の作成

Antigravityで以下を実行:

```
.agent/workflows フォルダを作成してください。
```

### Step 2: team.md の作成

以下の内容をAntigravityに伝えて、ワークフローファイルを作成させる。

**Antigravityへのコピペ用**:

```
.agent/workflows/team.md を作成してください。
以下の内容で実装してください。

---

【目的】
/team で起動する一気通貫開発ワークフロー
非エンジニアでも迷わず進められるように、AIから質問する形式

【フロー】
Step 1: 要件定義
- skills/requirements-gathering/SKILL.md を参照して実行
- フォルダ名を提案し、承認後に作成
- 完了後「設計に進みますか？」と確認

Step 2: 設計
- skills/design/SKILL.md を参照して実行
- 完了後「実装に進みますか？」と確認

Step 3: 実装
- docs/DESIGN.md を読み込み、設計に基づいて実装
- 完了後「レビューしますか？」と確認

Step 4: レビュー
- skills/review/SKILL.md を参照して実行
- 完了後「ドキュメントを作成しますか？」と確認

Step 5: ドキュメント
- skills/documentation/SKILL.md を参照して実行
- 完了後「振り返りをしますか？」と確認

Step 6: 振り返り
- skills/retrospective/SKILL.md を参照して実行

【完了時】
作成されたファイル一覧を表示して終了

【途中終了】
「ここで終わり」と言われたら、そこまでの成果物を保存して終了

---
```

### Step 3: 各スキルの作成

各スキルの詳細な作成方法は、個別記事を参照。

| スキル | 作成方法 |
|--------|----------|
| 要件定義 | [詳細記事](antigravity-requirements-skill) |
| 設計 | [詳細記事](antigravity-design-skill) |
| レビュー | [詳細記事](antigravity-review-skill) |
| ドキュメント | [詳細記事](antigravity-documentation-skill) |
| 振り返り | [詳細記事](antigravity-retrospective-skill) |

### Step 4: 動作確認

```
/team
```

と入力して、ワークフローが起動することを確認。

## つまずきポイント

### 1. ワークフローが自然言語で起動しない

**問題**: 「開発始めて」と言っても `/team` が起動しない。

**原因**: **ワークフローはスラッシュコマンドでのみ起動する**。これはAntigravityの仕様。

**解決策**: `/team` と明示的に入力する。

### 2. 「/team が見つかりません」エラー

**問題**: `/team` と入力してもワークフローが起動しない。

**原因**: `.agent/workflows/team.md` が存在しない。

**解決策**: Step 2 の手順でファイルを作成する。

### 3. スキルファイルが見つからない

**問題**: ワークフロー実行時に「スキルが見つかりません」エラー。

**原因**: スキルファイルが正しい場所にない。

**解決策**: 各スキルの詳細記事を参照して、正しいパスにファイルを作成する。

## 他のプロジェクトで再利用する方法

一度作成したワークフローとスキルは、フォルダをコピーすれば再利用できる。

```
# コピー元
projects/
├── .agent/
└── skills/

# コピー先（新しいプロジェクト）
new-project/
├── .agent/        ← コピー
└── skills/        ← コピー
```

**非エンジニアの個人開発でも、一度仕組みを作れば何度でも使える。**

## まとめ

- **非エンジニアでも個人開発できる** — AIが質問してくれるから
- `/team` で一気通貫開発 — 要件定義から振り返りまで
- **フォルダ名もAIが提案** — 命名センス不要
- **何を伝えればいいか迷わない** — 質問に答えるだけ
- ワークフローはデフォルトでは存在しない — 本記事の手順で作成が必要

**AIを使いこなすのに「完璧な指示」は要らない。AIに質問してもらえばいい。**
