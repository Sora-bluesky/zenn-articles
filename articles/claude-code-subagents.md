## はじめに

第1部で「コンテキストが埋まると精度が落ちる」と解説した。

では、大量のファイルを読む必要がある「調査」はどうすればいいのか？

答えは、**サブエージェントに委譲する**。

サブエージェントは別のコンテキストで動き、結果だけを報告してくれる。メインの会話を汚さずに、調査や検証ができる。

この記事では、サブエージェントの活用方法を解説する。

## サブエージェントとは

サブエージェントは、メインのClaude（普段会話している相手）から呼び出される「別のClaude」。

```
+---------------------------+
|  メインエージェント       |  ← 普段の会話相手
|  （Claude Code 本体）     |
+---------------------------+
       |
       | 「この調査をお願い」
       ↓
+---------------------------+
|  サブエージェント         |  ← 別のコンテキストで動く
|  （調査を担当）           |
+---------------------------+
       |
       | 「調査結果だけ報告」
       ↓
+---------------------------+
|  メインエージェント       |
|  （要約を受け取る）       |
+---------------------------+
```

**ポイント**：
- サブエージェントは**別のコンテキストウィンドウ**で動く
- 調査で100ファイル読んでも、メインのコンテキストは消費しない
- メインには**結果の要約だけ**が返ってくる

---

## なぜサブエージェントが有効なのか

### 調査をメインで行う場合

```
あなた：「認証システムの仕組みを調べて」
Claude：（50ファイルを読む）
Claude：「調査結果は...」

→ メインのコンテキストが50ファイル分消費される
→ その後の実装でコンテキストが不足しがち
```

### 調査をサブエージェントに委譲する場合

```
あなた：「サブエージェントを使って認証システムの仕組みを調べて」
サブエージェント：（別のコンテキストで50ファイルを読む）
メインのClaude：「調査結果の要約を受け取りました」

→ メインのコンテキストは要約分だけ
→ 実装に十分なコンテキストが残る
```

## 基本的な使い方

### 方法1：その場で指示する

```
サブエージェントを使って、認証システムがトークンリフレッシュを
どう処理しているか調べて。既存のOAuthユーティリティがあれば報告して。
```

Claudeが自動的にサブエージェントを使って調査し、結果を報告してくれる。

### 方法2：カスタムサブエージェントを定義する

よく使う調査パターンは、サブエージェントとして定義しておく。

ファイル：`.claude/agents/security-reviewer.md`
```markdown
---
name: security-reviewer
description: セキュリティの脆弱性をレビュー
tools: Read, Grep, Glob, Bash
model: opus
---
あなたはシニアセキュリティエンジニアです。
以下の観点でコードをレビューしてください：

- インジェクション脆弱性（SQL、XSS、コマンドインジェクション）
- 認証・認可の欠陥
- コード内のシークレットや認証情報
- 安全でないデータ処理

具体的な行番号と修正案を提示してください。
```

使い方：
```
security-reviewer サブエージェントを使って、この変更をレビューして
```

## 活用パターン

### パターン1：コードベースの調査

新しいプロジェクトで、仕組みを理解したい時。

```
サブエージェントを使って、このプロジェクトの認証フローを調べて。
特にセッション管理とトークンリフレッシュの仕組みを説明して。
```

### パターン2：実装後のレビュー

自分（Claude）が書いたコードを、別の視点でチェック。

```
サブエージェントを使って、今実装したコードのエッジケースを確認して
```

「自分が書いたコードに甘くなる」問題を、別のコンテキストでチェックすることで軽減できる。

### パターン3：複数の観点で同時調査

複数のサブエージェントを並列で動かすこともできる。

```
サブエージェントを2つ使って：
1. セキュリティ観点でレビュー
2. パフォーマンス観点でレビュー
```

## サブエージェントの設定項目

カスタムサブエージェントで設定できる項目。

| 項目 | 説明 | 例 |
|------|------|-----|
| `name` | サブエージェントの名前 | `security-reviewer` |
| `description` | 何をするか | `セキュリティ脆弱性をレビュー` |
| `tools` | 許可するツール | `Read, Grep, Glob` |
| `disallowedTools` | 禁止するツール | `Write, Edit` |
| `model` | 使用するモデル | `opus`, `sonnet`, `haiku` |

### 読み取り専用のサブエージェント

レビュー担当が勝手にコードを変更しないように：

```markdown
---
name: code-reviewer
tools: Read, Grep, Glob
disallowedTools: Write, Edit, NotebookEdit
---
```

`disallowedTools` で書き込み系ツールを禁止すると、読み取り専用になる。

## グローバルに配置する

一度作ったサブエージェントを全プロジェクトで使いたい場合：

**Windows（PowerShell）:**
```powershell
$dest = "$env:USERPROFILE\.claude\agents"
New-Item -ItemType Directory -Force -Path $dest
Copy-Item ".\.claude\agents\*.md" $dest
```

**配置場所の違い**：

| 場所 | パス | 用途 |
|------|------|------|
| ローカル | `.claude/agents/` | そのプロジェクトだけ |
| グローバル | `~/.claude/agents/` | 全プロジェクト共通 |

## サブエージェント vs 並列セッション

似たようなことを「並列セッション」でもできる。違いは：

| | サブエージェント | 並列セッション |
|--|-----------------|---------------|
| 起動 | 自動（Claudeが呼び出す） | 手動（自分でターミナルを開く） |
| 結果の受け渡し | 自動（要約が返る） | 手動（自分でコピペ） |
| コスト | 追加分だけ | 2倍 |
| 用途 | 一時的な調査・検証 | 継続的な並列作業 |

**使い分け**：
- 「ちょっと調べてきて」→ サブエージェント
- 「別の作業を並行で進めて」→ 並列セッション

## 注意点

### コンテキストは引き継がない

サブエージェントは新しいコンテキストで始まる。メインの会話の流れは知らない。

```
❌ 「さっき話したファイルを調べて」
✅ 「src/auth/session.ts を調べて」
```

具体的なファイルパスや情報を渡す必要がある。

### 結果は要約される

サブエージェントが100ファイル読んでも、返ってくるのは要約。

詳細が必要な場合は、追加で質問するか、範囲を絞って再調査させる。

## まとめ

| ポイント | 内容 |
|---------|------|
| 目的 | 調査でメインのコンテキストを消費しない |
| 使い方 | 「サブエージェントを使って〇〇を調べて」 |
| カスタム | `.claude/agents/` に定義ファイルを作成 |
| 読み取り専用 | `disallowedTools: Write, Edit` |
| vs 並列 | 一時的な調査→サブエージェント、並行作業→並列セッション |

**覚えておくこと**：

> コンテキストは最も重要なリソース。
> 調査はサブエージェントに任せて、メインは実装に集中する。
