## はじめに

Claude Codeを使うたびに、同じ説明を繰り返していないだろうか。

- 「コミットメッセージは日本語で書いて...」
- 「テストを通してから変更して...」
- 「このフォルダ構成は...」

毎回これを説明するのは面倒だし、説明し忘れると想定外の動作をされる。

**CLAUDE.md** は、この問題を解決する。一度書いておけば、Claudeが毎回自動で読んでくれる。

この記事では、CLAUDE.mdの書き方と、やってはいけないことを解説する。

## CLAUDE.md とは

CLAUDE.mdは、Claudeが会話の最初に自動で読むMarkdownファイル。プロジェクトのルール、個人の好み、組織の標準など、**コードを見ただけでは分からない情報**をClaudeに伝えるために使う。

配置場所によってスコープが変わる（詳しくは[配置場所](#claude.md-の配置場所)で解説する）。

---

## 始め方：`/init` コマンド

ゼロから書くのは大変なので、`/init` コマンドで雛形を生成する。

```
/init
```

Claudeがプロジェクトを分析して、ビルドシステム、テストフレームワーク、コードパターンなどを検出し、CLAUDE.mdの雛形を作ってくれる。

:::message
**ポイント**
`/init` で生成された内容をそのまま使うのではなく、自分のプロジェクトに合わせて調整する。既にCLAUDE.mdがある場合は上書きせず、改善提案をしてくれる。
:::

---

## 何を書くべきか

### 書くべきもの

| カテゴリ | 例 |
|---------|-----|
| Claudeが推測できないコマンド | `npm run dev:local`（独自スクリプト） |
| デフォルトと異なるコードスタイル | 「セミコロンは付けない」「シングルクォートを使う」 |
| テストの実行方法 | `npm test -- --watch` |
| リポジトリのルール | 「ブランチ名は feature/xxx 形式」「PRは日本語で書く」 |
| プロジェクト固有の設計判断 | 「状態管理はReduxではなくZustandを使う」 |
| 開発環境の癖 | 「.env.local が必要」「Docker が必要」 |
| よくある落とし穴 | 「このAPIは非推奨。代わりに〇〇を使う」 |

僕の場合、最初は「コミットメッセージは英語で」「日本語で応答して」の2行だけだった。それだけでも毎回説明する手間が消えて、体感が変わった。

その後、Claude Codeを開発したAnthropicのエンジニアBoris Cherny氏がXで公開したワークフローを参考に、自分のグローバルCLAUDE.mdを育てていった。

https://x.com/bcherny/status/2017742741636321619

Boris氏の原典は6カテゴリ・30行超だが、自分に必要なエッセンスだけ残して9行にした：

```markdown
## ワークフロー原則
- 非自明なタスクは Plan Mode で始める
- うまくいかないときは立ち止まって再計画する
- タスク完了前に必ず検証する
- サブエージェントを積極活用してメインコンテキストを保護する
- ユーザーに修正されたら、Auto Memoryに学びを記録する

## 品質基準
- 変更は最小限・最もシンプルな方法で
- 根本原因を特定する。一時的な修正で済ませない
- 機能追加とリファクタリングは別コミット
- 自分の出力を批判的に評価してから提示する
```

**CLAUDE.mdは短いほど守られる**ので、最初から完璧を目指さず、Claudeが間違えるたびに1行ずつ追加していくのがコツだ。

### 書くべきでないもの

| カテゴリ | 理由 |
|---------|------|
| コードを読めば分かること | 冗長。コンテキストを消費するだけ |
| リンターやフォーマッターで強制できること | ESLint、Prettierなどのツールに任せる |
| 言語の標準的な規約 | Claudeは既に知っている |
| 詳細なAPIドキュメント | リンクで済ませる |
| 頻繁に変わる情報 | メンテナンスが大変 |
| ファイルごとの説明 | Claudeはコードを読める |
| 長い説明やチュートリアル | CLAUDE.mdは指示書であって教科書ではない |
| 「きれいなコードを書け」のような自明なこと | 書かなくてもやる |

## 実例：シンプルなCLAUDE.md

以下はJavaScriptプロジェクトの例だが、どの言語でも考え方は同じ。

```markdown
# コードスタイル
- ES Modules（import/export）を使う。CommonJS（require）は使わない
- インポートは分割代入を使う（例：import { foo } from 'bar'）

# ワークフロー
- コード変更後は必ず型チェックを実行する
- テストは個別に実行する（全体実行は遅いため）

# コマンド
- 開発サーバー: npm run dev
- 型チェック: npm run typecheck
- テスト（単体）: npm test -- path/to/test.ts
```

**ポイント**：
- Markdownの見出し（`#`）と箇条書き（`-`）で構造化する。Claudeも人間と同じで、整理された文章のほうが正確に読める
- 具体的に書く。「コードをきれいに」ではなく「インデントは2スペース」
- Claudeが間違えそうなことだけ書く

## CLAUDE.md の配置場所

CLAUDE.mdは複数の場所に配置でき、それぞれ役割が違う。

| スコープ | 配置場所 | 何を書くか | 共有範囲 |
|---------|---------|-----------|---------|
| **個人（全体）** | `~/.claude/CLAUDE.md`（※） | 自分の好み：応答言語、コードスタイルの癖 | 自分だけ（全プロジェクト共通） |
| **プロジェクト** | `./CLAUDE.md` または `./.claude/CLAUDE.md` | チームの約束事：設計方針、コマンド、ワークフロー | チーム（Git管理推奨） |
| **個人（プロジェクト）** | `./CLAUDE.local.md` | 自分だけの設定：テスト用URL、サンドボックス情報 | 自分だけ（.gitignoreに追加） |

※ `~` はホームフォルダ（Windowsなら `C:\Users\ユーザー名\`）を指す記号。

**ポイント**：プロジェクトのCLAUDE.mdはチームとGitで共有するので、**個人の好みではなくプロジェクトの標準**を書く。「日本語で応答して」のような個人の好みは `~/.claude/CLAUDE.md` に、「ブランチ名はfeature/xxx形式」のようなチームルールはプロジェクトのCLAUDE.mdに書く。

**読み込み順**：個人（全体）→ プロジェクト → 個人（プロジェクト）の順で読み込まれる。子ディレクトリのCLAUDE.mdは、Claudeがそのフォルダ内のファイルを読んだときに自動で読み込まれる。

## 他のファイルを参照する

`@` 記法で他のファイルを参照できる。

```markdown
プロジェクトの概要は @README.md を参照。
npm コマンドは @package.json を参照。

# 追加の指示
- Git ワークフロー: @docs/git-workflow.md
- 個人設定: @~/.claude/my-overrides.md
```

これにより、CLAUDE.md自体を短く保ちつつ、必要な情報を参照できる。相対パスは**作業ディレクトリではなく、書いたファイルの場所**が基準になる点に注意。参照先からさらに別のファイルを参照する「再帰インポート」も可能で、**最大5階層**まで辿れる。

:::message
**初回だけ承認が必要**
プロジェクトで初めて `@` インポートを使うと、承認ダイアログが表示される。一度許可すれば次回以降は自動で読み込まれる。
:::

## CLAUDE.mdの限界を知っておく

CLAUDE.mdは**コンテキストであり、強制的な設定ではない**。Claudeは「できるだけ従おうとする」が、100%従う保証はない。

この前提を踏まえた上で、遵守率を上げるコツを見ていこう。

---

## よくある失敗：長すぎるCLAUDE.md

### 問題

CLAUDE.mdが長すぎると、**Claudeは内容を無視する**。

重要なルールが大量の文章に埋もれ、結果として守られない。

### 対策

各行について問いかける：

> 「この行を削除したら、Claudeは間違えるか？」

答えが「No」なら、削除する。あるいは「編集後に必ずlintを実行」のような**毎回確実に実行したい処理**は、CLAUDE.mdではなく[Hooks](claude-code-auto-format-hooks)に変換するほうが確実だ。

### 目安

公式ドキュメントでは **1ファイルあたり200行以下** が推奨されている。ただし、短いほど遵守率は上がる。

- 理想：50行以内（筆者の体感）
- 許容：200行以内（公式推奨上限）
- 危険：200行超（重要なルールが埋もれる）

長くなってきたら、次のセクションで紹介する `.claude/rules/` に分割する。

---

## 長くなったら `.claude/rules/` に分割する

CLAUDE.mdが長くなってきたら、`.claude/rules/` ディレクトリにルールを分割できる。

```
プロジェクトフォルダ/
├── .claude/
│   ├── CLAUDE.md           ← メインの指示（短く保つ）
│   └── rules/
│       ├── code-style.md   ← コードスタイル
│       ├── testing.md      ← テスト規約
│       └── security.md     ← セキュリティ要件
└── ...
```

**特徴**：
- `.md` ファイルを再帰的に検出するので、`frontend/`、`backend/` のようなサブフォルダ構成もOK
- 起動時にCLAUDE.mdと同じタイミングで読み込まれる
- CLAUDE.md本体を短く保ちながら、ルールを整理できる

この構造は **Progressive Disclosure（段階的な情報開示）** という考え方に基づいている。CLAUDE.mdに全てを詰め込むのではなく、情報を3層に分けて「必要な時に必要なだけ」読み込ませる。

| 層 | 何を置くか | いつ読まれるか |
|----|-----------|---------------|
| **CLAUDE.md** | 全体方針、ビルドコマンド、ワークフロー | 毎回 |
| **rules/** | コードスタイル、テスト規約などトピック別 | 毎回（起動時に自動読み込み） |
| **Skills** | 特定の作業手順、API仕様など | 必要な時だけ |

CLAUDE.mdは「目次」のように短く保ち、詳細はrules/やSkillsに委譲する。

---

## うまくいかない時のデバッグ

### まず `/memory` で読み込み状況を確認する

Claude Code上で `/memory` と入力すると、現在読み込まれているCLAUDE.mdとrulesファイルの一覧が表示される。ファイルがリストにない場合、Claudeはそれを見ていない。

### Claudeがルールを無視する

**原因1**：CLAUDE.mdが長すぎる
→ 短くする。本当に必要な情報だけ残す。

**原因2**：表現が曖昧
→ 具体的に書く。「きれいなコードを書け」ではなく「関数は20行以内にする」。

**原因3**：強調が足りない
→ 重要なルールには `IMPORTANT` や `YOU MUST` を付ける。公式ドキュメントでもこの方法で**遵守率が向上する**と明記されている。

**原因4**：ルール同士が矛盾している
→ 複数のCLAUDE.mdやrules/ファイルに矛盾する指示があると、Claudeは**どちらか片方を勝手に選ぶ**。定期的に見直して、古いルールや重複するルールを整理する。

### Claudeが同じ質問を繰り返す

**原因**：CLAUDE.mdに答えが書いてあるが、表現が分かりにくい
→ より直接的な表現に書き換える。

### `/compact` したら指示が消えた？

CLAUDE.mdの内容は `/compact`（会話の圧縮）を**完全に生き残る**。圧縮後、Claudeはディスクからファイルを再読み込みする。

もし `/compact` 後に指示が消えたなら、それはCLAUDE.mdではなく**会話の中だけで伝えた指示**だった可能性が高い。繰り返し使う指示はCLAUDE.mdに書いておこう。

## 応用：チームで共有する

CLAUDE.mdをGitにコミットすると、チーム全員で共有できる。

```
project/
├── CLAUDE.md           ← チーム共有（Gitにコミット）
├── CLAUDE.local.md     ← 個人設定（.gitignoreに追加）
└── ...
```

`.gitignore`:
```
CLAUDE.local.md
```

**チーム共有のメリット**：
- 新メンバーがClaude Codeを使い始めても、プロジェクトのルールが適用される
- 「このプロジェクトでは〇〇を使う」という暗黙知を明文化できる

## Skills（スキル）との使い分け

CLAUDE.mdは**毎回読み込まれる**。

一方、**Skills**は必要な時だけ読み込まれる。

| | CLAUDE.md | Skills |
|--|-----------|--------|
| 読み込み | 毎回 | 必要な時だけ |
| 用途 | 常に適用するルール | 特定の作業に必要な知識 |
| 例 | コードスタイル、テストコマンド | APIの使い方、特定機能の実装手順 |

**使い分け**：
- 「どんな作業でも守ってほしいルール」→ CLAUDE.md
- 「特定の作業でだけ必要な知識」→ Skills

:::message
**補足：auto memory（自動メモリ）**
CLAUDE.mdとは別に、Claude自身が学んだことを自動で記録する「auto memory」という仕組みもある。こちらはClaudeが勝手にメモを取る機能で、手動で書く必要はない。`/memory` コマンドで確認・編集できる。
:::

## まとめ

| ポイント | 内容 |
|---------|------|
| 目的 | コードを見ただけでは分からない情報を伝える |
| 始め方 | `/init` で雛形を生成 |
| 書くべきもの | 独自コマンド、スタイルルール、落とし穴 |
| 書くべきでないもの | コードを読めば分かること、自明なこと |
| 長さ | 200行以下（公式推奨）。短いほど遵守率が上がる |
| 分割 | 長くなったら `.claude/rules/` に分割 |
| 配置 | プロジェクトルートに置いてGit管理 |
| 確認 | `/memory` で読み込み状況を確認 |

**覚えておくこと**：

> CLAUDE.mdが長すぎると、Claudeはルールを無視する。
> 短く、具体的に、本当に必要なことだけ書く。
