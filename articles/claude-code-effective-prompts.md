## はじめに

同じことをClaudeに頼んでも、指示の仕方で結果が大きく変わる。

**曖昧な指示**：
```
ログイン機能を作って
```

**具体的な指示**：
```
src/auth/ にログイン機能を追加して。
既存の AuthProvider パターンに従って実装して。
セッションタイムアウト後のリフレッシュ処理も含めて。
実装後に auth.test.ts のテストが通ることを確認して。
```

:::message
上の例は公式ドキュメントからの引用で、ファイルパスは自分のプロジェクトに合わせて読み替えてほしい。自分のプロジェクトの場合は「ログインフォームを作って。メールアドレスとパスワードを入力できるようにして」のように書けばいい。
:::

前者は「何となくそれっぽいもの」ができる。後者は「欲しいもの」ができる。

この記事では、Claudeに的確に伝えるためのプロンプト術を解説する。

## 基本原則：具体的であるほど良い

Claudeは意図を推測できるが、読心術は使えない。

具体的に伝えるほど、修正のやり取りが減り、結果的に速く終わる。

### Before / After 比較

:::message
以下のプロンプト例はClaude Code公式ドキュメントからの引用。ファイルパスは自分のプロジェクトに合わせて読み替えてほしい。
:::

| Before（曖昧） | After（具体的） |
|----------------|-----------------|
| テストを追加して | foo.py にテストを追加して。ユーザーがログアウトしている場合のエッジケースをカバーして。モックは使わないで |
| なぜこのAPIは変なの？ | ExecutionFactory の git 履歴を見て、このAPIがどうしてこうなったか要約して |
| カレンダーを追加して | ホームページの既存ウィジェットの実装パターンを見て（HotDogWidget.php が良い例）。それに従ってカレンダーウィジェットを実装して。月選択とページネーション機能を付けて |
| ログインのバグを直して | ユーザーがセッションタイムアウト後にログインできないと報告している。src/auth/ の認証フロー、特にトークンリフレッシュを確認して。問題を再現するテストを書いてから修正して |

## 4つの指定要素

「何を伝えればClaude が迷わないか」を分解すると、4つに落ち着く。

### 1. スコープ（どこを）

```
src/components/Button.tsx の...
```

ファイルやフォルダを明示する。「ボタン」ではなく「src/components/Button.tsx」。

### 2. 要件（何を）

```
...disabled 状態のスタイルを追加して。
グレーアウトして、クリックできないようにして。
```

具体的な仕様を伝える。

### 3. 制約（どうやって）

```
既存の theme.ts の色定義を使って。
新しい色は追加しないで。
```

使うべきもの、使ってはいけないものを明示する。

### 4. 検証（どう確認する）

```
Button.test.tsx のテストが通ることを確認して。
```

完了条件を明示する（第2部で解説した内容）。

## 探索 → 計画 → 実装 のワークフロー

いきなりコードを書かせると、間違った方向に進むことがある。

公式が推奨するワークフローは4段階。

### Step 1: 探索（Plan Mode）

まず、コードを読んで理解させる。

```
Plan Mode に切り替えて。
src/auth/ を読んで、セッション管理とログインの仕組みを理解して。
環境変数でシークレットをどう管理しているかも見て。
```

Plan Mode では、Claudeはファイルを読むだけで変更しない。

### Step 2: 計画（Plan Mode）

理解したら、計画を立てさせる。

```
Google OAuth を追加したい。
どのファイルを変更する必要がある？
セッションの流れはどうなる？
計画を作って。
```

:::message
**Ctrl + G で計画を編集**
計画が表示されたら、Ctrl + G でテキストエディタで開いて直接編集できる。
:::

### Step 3: 実装（Normal Mode）

計画に問題がなければ、実装させる。

```
Normal Mode に切り替えて。
計画に従って OAuth フローを実装して。
コールバックハンドラのテストを書いて、テストスイートを実行して、失敗があれば修正して。
```

### Step 4: コミット

完了したら、コミットさせる。

```
分かりやすいメッセージでコミットして、PR を作って。
```

---

## いつ計画をスキップするか

Plan Mode は強力だが、オーバーヘッドもある。

**計画が必要な場合**：
- アプローチが不明確
- 複数ファイルを変更する
- 変更するコードに不慣れ

**計画をスキップしていい場合**：
- タイポの修正
- ログの追加
- 変数名の変更
- 差分を一言で説明できる変更

## リッチなコンテンツを与える

テキストだけでなく、様々な形式で情報を渡せる。

### ファイルを参照する：`@`

```
@src/auth/session.ts を見て、セッション管理の仕組みを説明して
```

`@` でファイルを参照すると、Claudeがそのファイルを読んでから回答する。

### 画像を貼り付ける

スクリーンショットやデザイン画像をそのまま貼り付けられる。

```
[画像を貼り付け]
このデザインを実装して
```

### URLを指定する

```
https://example.com/api-docs を参照して、認証の仕組みを理解して
```

:::message
`/permissions` で頻繁にアクセスするドメインを許可リストに追加しておくと、毎回確認されなくて済む。
:::

### データをパイプで渡す

```bash
cat error.log | claude "このエラーの原因を分析して"
```

ファイルの内容を直接渡せる。

## 軌道修正のテクニック

Claudeが間違った方向に進んだ時の対処法。

### Esc キーで中断

途中で「違う」と思ったら、`Esc` で止める。コンテキストは保持されるので、すぐに別の指示を出せる。

### `/rewind` で巻き戻す

変更を取り消したい時は `/rewind`（または `Esc` 2回）。

### 履歴を編集する

過去の入力を編集して、別のルートを試せる。「さっきの指示をこう変えていたら」を試行できる。

### 2回修正しても直らなかったら `/clear`

修正を繰り返すと、失敗した試行錯誤がコンテキストに残り、さらに混乱する。

2回修正しても直らなかったら、`/clear` して最初から。学んだことを活かして、より良い最初の指示を書く。

## 曖昧な指示が有効な場合

常に具体的である必要はない。

**探索的な質問**：
```
このファイルの改善点は？
```

何を聞くべきか分からない時、オープンな質問が発見につながることがある。

**アイデア出し**：
```
この機能を実装する方法を3つ提案して
```

選択肢を出させてから、具体的な方向を決める。

---

## インタビュー形式で要件を詰める

大きな機能を作る時は、最初にClaudeにインタビューさせる。

```
[簡単な説明] を作りたい。
詳しくインタビューして。

技術的な実装、UI/UX、エッジケース、トレードオフについて聞いて。
自明な質問はしないで、考えていなかった難しい部分を掘り下げて。

全てカバーしたら、完全な仕様を SPEC.md に書いて。
```

Claudeが質問してくれるので、自分が考えていなかった点に気づける。

仕様ができたら、新しいセッションを開始して実装に集中する。

## まとめ

| ポイント | 内容 |
|---------|------|
| 基本原則 | 具体的であるほど良い |
| 4要素 | スコープ、要件、制約、検証 |
| ワークフロー | 探索 → 計画 → 実装 → コミット |
| 軌道修正 | Esc で中断、/rewind で巻き戻し |
| 失敗したら | 2回修正しても直らなければ /clear |

**覚えておくこと**：

> 曖昧な指示に修正を重ねるより、
> /clear して具体的な指示を最初から書く方が速い。
