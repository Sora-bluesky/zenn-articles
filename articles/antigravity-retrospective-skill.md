## はじめに

ツールが完成した。上司にも報告した。

やった、終わった！

...でも、ふと思う。

---

「次に同じようなツール作る時、また3日悩むのかな...」

---

「『レシート撮影』って言ったらスマホアプリ作られたやつ、また起きそう...」

---

「今回うまくいったこと、何だったっけ？」

...もう忘れてる。

---

非エンジニアの個人開発で五番目につらいのは、**「同じ失敗を繰り返す」**こと。

本記事では、**AIに質問に答えるだけで、良かったこと・問題点・次やることを整理するスキル**の詳細を解説する。

## 前提条件

- Google Antigravity インストール済み
  - まだの場合 → [Antigravityインストールガイド](antigravity-windows-install-guide)
- プロジェクトが完了している
- プログラミング経験は不要

## 解決策：振り返りスキル

### コンセプト

**「振り返りして」と言うだけで、良かったこと・問題点・次やることが整理される。**

```
ユーザー: 振り返りして

AI: いくつか質問させてください。

    Q1: 満足度は？
    A. とても満足
    B. 満足
    C. やや不満
    D. 不満

（5つの質問に答える）

AI: 振り返りレポートを生成しました！

    ## 良かった点
    - AIの提案が的確だった
    - 設計書を先に作ったのが良かった

    ## 問題点
    - 「レシート撮影」の意味が伝わらなかった
    - 実装で苦労した

    ## 次やること
    - 最初に「GASで作りたい」と伝える
    - 設計書を先に作るのは続ける
```

### 「良かった点・問題点・次やること」とは

シンプルな振り返りのフレームワーク。

| 項目 | 意味 | 例 |
|------|------|-----|
| **良かった点** | うまくいったこと。次もやりたい | 「設計書を先に作ったのが良かった」 |
| **問題点** | 困ったこと。同じ失敗をしたくない | 「『レシート撮影』でスマホアプリになった」 |
| **次やること** | 次回試したいこと | 「最初に『GASで作りたい』と伝える」 |

:::message
これは「KPT」と呼ばれる振り返り手法。
Keep（続けたい）、Problem（問題）、Try（試したい）の頭文字だ。
でも、非エンジニアには「良かった点・問題点・次やること」の方がわかりやすい。
:::

### 5つの質問

| # | 質問 | 形式 | なぜ聞くのか |
|---|------|------|-------------|
| 1 | 満足度は？ | 選択式 | 全体の評価を把握 |
| 2 | 一番大変だったのは？ | 選択式 | 問題点を特定 |
| 3 | 予定より時間がかかった？ | 選択式 | スケジュールの振り返り |
| 4 | 特に良かった点は？ | 自由回答 | 良かった点の材料 |
| 5 | 次回改善したい点は？ | 自由回答 | 次やることの材料 |

**選択式 + 自由回答の組み合わせで、効率的に振り返りができる。**

## 実装方法

### Antigravityに指示を出す

```
skills/retrospective/SKILL.md を作成してください。
フォルダがなければ作成してください。

【解決したい悩み】
非エンジニアの個人開発で、
1. 同じ失敗を繰り返す
2. 今回うまくいったことを忘れる
3. 次に活かせない

【自動参照する情報】
- 要件サマリー
- docs/DESIGN.md
- docs/REVIEW.md
- README.md

【質問フロー（この順序を必ず守ること）】
Q0: プロジェクト情報の確認（必須・最初に行う）
Q1: プロジェクト全体の満足度は？（選択式）
    A. とても満足（期待以上）
    B. 満足（期待通り）
    C. やや不満（改善の余地あり）
    D. 不満（多くの課題が残った）

Q2: 一番大変だったのはどこですか？（選択式）
    A. 何を作るか決めるところ（要件定義）
    B. どう作るか決めるところ（設計）
    C. 実際に作るところ（実装）
    D. チェック・修正するところ（レビュー）
    E. ドキュメントを書くところ

Q3: 予定より時間がかかりましたか？（選択式）
    A. 予定より早く終わった
    B. だいたい予定通り
    C. 少し遅れた
    D. かなり遅れた

Q4: 特に良かった点は何ですか？（自由回答）
    例: 設計書を先に作ったのが良かった、AIの提案が的確だった

Q5: 次回改善したい点は何ですか？（自由回答）
    例: 最初に「GASで作りたい」と伝える、テストを先に考える

【出力形式】
- YAML frontmatter を含める（学びの継承スキルとの連携用）
- 「良かった点・問題点・次やること」形式
- 具体的なアクション3つ
- docs/RETROSPECTIVE.md に保存
- .insights/retrospectives/ にもコピー

【制約事項】
- 質問の省略禁止: Q0〜Q5 はすべて実行すること
- 順序変更禁止: 必ず Q0 → Q1 → Q2 → Q3 → Q4 → Q5 の順で実行
- 1問ずつ対話: 複数の質問を一度に聞かない
- YAML frontmatter必須: 学びの継承スキルとの連携のため
- 専門用語を使わない
- 「KPT」ではなく「良かった点・問題点・次やること」と表現する
```

**Q2: 一番大変だったところ**

```

A. 何を作るか決めるところ（要件定義）
B. どう作るか決めるところ（設計）
C. 実際に作るところ（実装）
D. チェック・修正するところ（レビュー）
E. ドキュメントを書くところ
```

**Q3: スケジュール**

```
AI: 予定より時間がかかりましたか？

A. 予定より早く終わった
B. だいたい予定通り
C. 少し遅れた
D. かなり遅れた

ユーザー: C
```

**Q4: 良かった点（自由回答）**

```
AI: 特に良かった点は何ですか？
    例: 設計書を先に作ったのが良かった、AIの提案が的確だった

ユーザー: 設計スキルで選択肢から選べたのが良かった。
         迷わなかった。
```

**Q5: 改善点（自由回答）**

```
AI: 次回改善したい点は何ですか？
    例: 最初に「GASで作りたい」と伝える、テストを先に考える

ユーザー: 最初に「レシート撮影」って言ったらスマホアプリを作られた。
         「GASでWebアプリ」って最初に言えばよかった。
```

**レポート生成**

```
AI: 振り返りレポートを生成しました！

---

# 振り返りレポート

## プロジェクト概要
- プロジェクト名: team-expense-assistant
- 満足度: ⭐ 満足

---

## 良かった点
- 設計スキルで選択肢から選べた → 迷わなかった
- 無料のツールだけで完成できた
- レビューで問題を事前に発見できた

---

## 問題点
- 要件定義で苦労した（Q2回答）
- 少し遅れた（Q3回答）
- 「レシート撮影」でスマホアプリを作られそうになった

---

## 次やること
- 最初に「GASで作りたい」と伝える
- 設計スキルの選択式は使い続ける
- 要件定義の時に技術の制約も伝える

---

## 具体的なアクション（3つ）
1. 次回は最初に「GASで、無料で」と伝える
2. 設計スキルの選択式は引き続き使う
3. 振り返りを毎回やる

---

docs/RETROSPECTIVE.md に保存しますか？

ユーザー: はい

AI: 保存しました ✅
```

## 非エンジニアの個人開発でよくある振り返り

| 問題点 | 次やること |
|--------|-----------|
| 「レシート撮影」でスマホアプリになった | 最初に「GASで作りたい」と伝える |
| 選択肢が多すぎて選べなかった | 設計スキルの選択式を使う |
| 3日経っても完成しなかった | 要件定義で「3つに絞る」を守る |
| 動くけど大丈夫かわからなかった | レビュースキルを毎回使う |
| フォルダ名で30分悩んだ | 要件定義スキルでAIに提案してもらう |

**次回の個人開発で、同じ失敗をしないための記録になる。**

## つまずきポイント

### 1. 自由回答が思いつかない

**問題**: 「良かった点は？」と聞かれても、すぐに出てこない。

**解決策**: スキルに「例」を表示するように設定している。例を参考に思い出せる。

### 2. 振り返りの時間がない

**問題**: 完成したら次のタスクに移りたい。振り返りは後回し。

**解決策**: ワークフローで最後に自動で振り返りフェーズに入る。流れで振り返れる。

## まとめ

- **非エンジニアでも振り返りできる** — AIが質問してくれる
- 5つの質問（選択式 + 自由回答）で効率的に情報収集
- 「良かった点・問題点・次やること」でシンプルに整理
- 次回の個人開発で同じ失敗をしない

**これでシリーズ完結。全体像は「AIへの指示が難しい」を解決する を参照。**
