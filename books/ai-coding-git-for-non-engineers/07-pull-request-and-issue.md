## この章のゴール

> この章を終えると、Pull Request（プルリクエスト）を使って変更をレビューに出せるようになり、Claude Code が自動で PR を作成する意味がわかるようになる。

## Pull Request とは何か

前章で、ブランチの変更を main にマージする方法を学んだ。しかし実際のプロジェクトでは、**いきなりマージするのではなく、「変更を見てもらってから」マージする**のが普通である。

この「変更を見てもらう」仕組みが **Pull Request（プルリクエスト、略称 PR）** である。

日常業務に例えるなら、こういう流れだ。

| 手順 | 日常業務                         | Git / GitHub               |
| ---- | -------------------------------- | -------------------------- |
| 1    | 企画書のドラフトを書く           | ブランチで作業してコミット |
| 2    | 上司に「見てください」と提出     | **Pull Request を作成**    |
| 3    | 上司がコメントや修正指示をくれる | **レビュー**               |
| 4    | 指摘を反映する                   | ブランチに追加コミット     |
| 5    | 「OK」の承認が出る               | **Approve（承認）**        |
| 6    | 正式版に反映                     | **マージ**                 |

### AI ツールと Pull Request

Claude Code に仕事を頼むと、多くの場合こう動く。

1. Claude Code がブランチを作る
2. コードを書いてコミットする
3. **Pull Request を自動で作成する**
4. あなたに「確認してください」と伝える

つまり、**あなたの役割は「AI が出した PR をレビューして、承認するかどうか判断すること」** である。コードの書き方を知らなくても、「何が変わったか」を確認して OK/NG を出せれば、AI と協働できる。

## Pull Request を作ってみる

### 前提

前章で作った `feature/add-about-page` ブランチ（または新しいブランチ）に、何かしらの変更がコミットされている状態を想定する。

### 手順 1: ブランチを GitHub に push する

1. 作業ブランチに切り替わっていることを確認（画面左下）
2. VS Code のターミナルで以下を実行

```bash
git push -u origin feature/add-about-page
```

または、「ソース管理」パネル →「ブランチの発行」をクリック。

### 手順 2: GitHub で Pull Request を作成する

1. ブラウザで GitHub のリポジトリページを開く
2. 黄色いバナーに「Compare & pull request」ボタンが表示されている → クリック
3. 以下を入力

| 項目     | 入力内容                                       |
| -------- | ---------------------------------------------- |
| タイトル | 変更内容を一言で（例: 「About ページを追加」） |
| 説明     | 変更の詳細や理由を書く                         |

4. 「Create pull request」をクリック

これで Pull Request が作成される。

### 手順 3: 変更内容を確認する

PR のページには「Files changed」タブがある。ここをクリックすると、**何がどう変わったかが一目でわかる**。

- 緑色の行 = 追加された内容
- 赤色の行 = 削除された内容

AI が書いたコードをレビューするときも、この画面を使う。

### AI が作った PR をレビューするコツ

プログラミングがわからなくても、以下のポイントを確認するだけで十分なレビューができる。

**1. 変更ファイルの数を見る**

「Files changed」の横に表示される数字が、変更されたファイルの数である。頼んだ内容に対して異常に多い場合は「余計なことをしていないか？」と確認する。

**2. 赤と緑のバランスを見る**

- 緑が多い = 新しいコードが追加された（機能追加）
- 赤が多い = 既存のコードが削除された（注意して確認）
- 赤と緑が同程度 = 書き換えが行われた

**3. 知っている単語を探す**

コードの中に、自分が指示した単語（例: 「山田太郎」「デザイナー」）が正しく反映されているか確認する。

**4. `.env` や秘密情報が含まれていないか**

ファイル一覧に `.env`、`password`、`secret`、`token`、`key` などの名前が含まれていたら要注意。第 9 章のトラブルシューティングを参照。

:::message
「コードが正しいかわからない」と不安になるかもしれないが、最初は「頼んだ通りのものができているか」「変なファイルが含まれていないか」だけチェックすれば十分である。
:::

### 手順 4: マージする

内容に問題がなければ、「Merge pull request」→「Confirm merge」をクリック。

これで、ブランチの変更が main に取り込まれる。

## Issue — やることリストとして使う

GitHub にはもう 1 つ便利な機能がある。**Issue（イシュー）**だ。

Issue は「やること」「気になること」「バグの報告」をチケットとして管理する機能である。

- 「ログイン画面のデザインを変えたい」→ Issue を作成
- 「このボタンを押すとエラーが出る」→ Issue を作成
- 「来週までにトップページを完成させる」→ Issue を作成

### AI ツールと Issue の関係

Claude Code に Issue の番号を伝えると、**Issue の内容を読み取って、その解決に取り組んでくれる**。

```
あなた: Issue #3 を解決して

Claude Code: Issue #3「About ページにプロフィール画像を追加」を確認しました。
  対応するブランチを作成して作業を開始します。
```

つまり、Issue を書くことは**AI への仕事の依頼書**を書くことに等しい。

### Issue の作り方

1. GitHub のリポジトリページで「Issues」タブをクリック
2. 「New issue」をクリック
3. タイトルと説明を入力
4. 「Submit new issue」をクリック

:::message
Issue のタイトルは「何をしてほしいか」を具体的に書くのがコツである。「デザインを直して」よりも「トップページのヘッダー色を青（#3B82F6）に変更する」のほうが、AI にも人間にも伝わりやすい。
:::

## 「やらかした！」復旧コーナー

### 「PR をマージした後に問題が見つかった」

GitHub の PR ページに「Revert」ボタンがある。これをクリックすると、マージの内容を打ち消す新しい PR が自動で作成される。

### 「PR のブランチをさらに修正したい」

PR はマージする前であれば、同じブランチに追加コミットするだけで、自動的に PR にも反映される。

1. ローカルで作業ブランチに切り替え
2. ファイルを修正
3. add → commit → push

これだけで、PR の「Files changed」が更新される。

## 裏で何が起きているか（CLI 解説）

Pull Request は Git のコマンドではなく、**GitHub の機能**である。CLI で操作するには GitHub CLI（`gh` コマンド）を使う。

```bash
# PR を作成
gh pr create --title "About ページを追加" --body "About ページの初期版を作成しました"

# PR の一覧を表示
gh pr list

# PR をマージ
gh pr merge 番号
```

Claude Code は内部的にこの `gh` コマンドを使って PR を自動作成している。
