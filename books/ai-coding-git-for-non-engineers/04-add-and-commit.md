## この章のゴール

> この章を終えると、VS Code の画面操作でファイルの変更を Git に記録でき、AI が書いたコードを「確定」する意味がわかるようになる。

## 「保存」と「記録」は違う

パソコンでファイルを `Ctrl + S` で保存するのは、あくまで「ファイルの中身を上書きする」だけである。Git の「記録」は、それとはまったく別の操作だ。

| 操作                     | 何が起きるか                                   | 元に戻せるか                   |
| ------------------------ | ---------------------------------------------- | ------------------------------ |
| ファイル保存（Ctrl+S）   | ファイルが上書きされる                         | Ctrl+Z で直前のみ              |
| **Git に記録（commit）** | **変更内容がスナップショットとして保存される** | **いつでも、何回前でも戻せる** |

Git の記録は「写真」に似ている。ある瞬間のプロジェクト全体を写真に撮って保存するイメージだ。何枚でも撮れるし、古い写真の状態に戻すこともできる。

## 2 つのステップ: add → commit

Git で変更を記録するには、 **2 つのステップ** が必要である。

### ステップ 1: 選ぶ（add = ステージング）

「今回の記録に含めたいファイル」を選ぶ操作である。

買い物に例えるなら、 **買い物カゴに商品を入れる** ようなものだ。カゴに入れただけでは、まだ購入は確定していない。

### ステップ 2: 確定する（commit）

選んだファイルをまとめて記録する操作である。

買い物で言えば、 **レジで精算する** ことに相当する。精算すると、「いつ、何を買ったか」が記録される。

![add と commit の流れ](/images/book-04-add-commit.png)

### なぜ 2 ステップなのか

「全部まとめて記録すればいいのに、なぜ選ぶ必要があるのか？」と思うかもしれない。

実務では、複数のファイルを同時に変更することがよくある。しかし、記録は **「意味のあるまとまり」ごとに分けたほうが、後から見返しやすい**。

例えば、「デザインの変更」と「バグの修正」を同時に行った場合、それぞれ別の記録にしておけば、後で「デザインだけ元に戻す」ことができる。

とはいえ、最初のうちは深く考えなくてよい。 **変更したファイルを全部選んで、まとめて記録する** だけで十分である。

### 3 つのエリアと add / commit の関係

第 3 章で紹介した「3 つのエリア」を、add / commit の操作と結びつけて見てみよう。

![M/A ステータスの遷移](/images/book-04-status-ma.png)

1. ファイルを編集・保存すると「作業ディレクトリ」に変更がある状態（M）
2. `add` で「ステージング」に移す（A）= 「次に記録に含める」と決める
3. `commit` で「リポジトリ」に確定（記録完了）

**AI コーディングとの関係:** Claude Code がファイルを編集した後、「変更を確認してください」と言ってくるのは、ステップ 2（add）とステップ 3（commit）をあなたに判断させるためである。

## VS Code で実際にやってみる

前の章で作った `hello.txt` を Git に記録してみよう。

### 手順 1: ファイルをステージングする（add）

1. VS Code の左側にある「ソース管理」アイコンをクリック
2. 「変更」セクションに `hello.txt` が表示されている
3. ファイル名の横にある **+** ボタン（ステージング）をクリック

ファイルが「変更」セクションから **「ステージされている変更」** セクションに移動する。これが「買い物カゴに入れた」状態である。

:::message
すべてのファイルをまとめてステージングしたい場合は、「変更」セクションのタイトル横にある **+** ボタンをクリックする。
:::

### 手順 2: コミットメッセージを書く

ステージングが終わったら、「この記録が何の変更なのか」を書く。これを **コミットメッセージ** と呼ぶ。

1. 「ソース管理」パネルの上部にあるテキストボックスにメッセージを入力
2. 例: `はじめてのファイルを追加`

コミットメッセージは、日本語でも英語でも構わない。ただし、 **何をしたかが一目でわかる** ように書くのがコツである。

#### 良いコミットメッセージの例

| 良い例                            | なぜ良いか                 |
| --------------------------------- | -------------------------- |
| `トップページのヘッダーを追加`    | 何をしたかが具体的         |
| `ログインボタンの色を青に変更`    | 変更箇所と内容が明確       |
| `README にプロジェクト概要を記載` | 対象ファイルと目的がわかる |

| 悪い例   | なぜ悪いか                 |
| -------- | -------------------------- |
| `修正`   | 何を修正したかわからない   |
| `更新`   | 何を更新したかわからない   |
| `作業中` | 後から見返したとき意味不明 |

#### AI コーディングでのコツ

Claude Code がコミットを行う場合、AI が自動でメッセージを生成してくれることが多い。その場合も、自分で内容を確認して「何の変更かわかるか？」を判断する習慣をつけておくとよい。

チームで使う場合は、メッセージの先頭にカテゴリをつける慣例がある。

```
feat: ログイン機能を追加     ← 新しい機能
fix: メールアドレスの検証を修正  ← バグ修正
docs: 利用規約ページを追加    ← ドキュメント
style: ヘッダーの余白を調整   ← 見た目の変更
```

最初のうちは気にしなくてよいが、慣れてきたら取り入れてみてほしい。

### 手順 3: コミットする

1. テキストボックスの上にある **チェックマーク（✓）** ボタンをクリック
2. またはキーボードで `Ctrl + Enter`（Mac は `Cmd + Enter`）

これで、変更が Git に記録された。

### 手順 4: 記録を確認する

ちゃんと記録されたか確認してみよう。

1. VS Code のターミナルを開く（「ターミナル」→「新しいターミナル」）
2. 以下のコマンドを入力

```bash
git log --oneline
```

こんな表示が出るはずだ。

```
a1b2c3d はじめてのファイルを追加
```

左側の `a1b2c3d` は **コミットハッシュ** と呼ばれる識別番号で、Git が自動的に付与する。各記録を一意に識別するためのものである。

## もう 1 つ記録してみる

理解を深めるために、もう 1 回記録してみよう。

1. `hello.txt` を開いて、内容を書き足す（例: 「Git の練習中」）
2. `Ctrl + S` で保存
3. 「ソース管理」パネルで `hello.txt` の横にある **+** をクリック（add）
4. メッセージを入力（例: `テキストを追記`）
5. **✓** ボタンをクリック（commit）

`git log --oneline` を実行すると、記録が 2 つに増えているのがわかる。

```
b4c5d6e テキストを追記
a1b2c3d はじめてのファイルを追加
```

新しい記録が上、古い記録が下に表示される。このように、Git は変更の **積み重ね** を記録していく。

## 「やらかした！」復旧コーナー

### 「間違えて add してしまった（ステージングを取り消したい）」

「ステージされている変更」セクションのファイル名の横にある **−**（マイナス）ボタンをクリックする。ファイルが「変更」セクションに戻り、ステージングが取り消される。

ファイルの内容自体は変わらないので安心してほしい。

### 「直前のコミットメッセージを間違えた」

直前のコミットメッセージだけを修正したい場合は、VS Code のターミナルで以下を実行する。

```bash
git commit --amend -m "正しいメッセージ"
```

:::message alert
`--amend` は **直前のコミットを書き換える** 操作である。まだ GitHub にアップロード（push）していない段階でのみ使うこと。push 済みのコミットを amend すると、トラブルの原因になる。
:::

## 裏で何が起きているか（CLI 解説）

| VS Code の操作             | CLI コマンド                      | 意味                       |
| -------------------------- | --------------------------------- | -------------------------- |
| **+** ボタン（1ファイル）  | `git add ファイル名`              | そのファイルをステージング |
| **+** ボタン（全ファイル） | `git add .`                       | すべての変更をステージング |
| **−** ボタン               | `git restore --staged ファイル名` | ステージングを取り消す     |
| **✓** ボタン               | `git commit -m "メッセージ"`      | ステージング内容を記録     |

### git diff — 「何が変わったか」を確認する

コミット前に「どこが変わったか」を確認したいときは `git diff` を使う。AI が大量のファイルを変更した場合に特に役立つ。

```bash
git diff
```

表示例:

```diff
diff --git a/hello.txt b/hello.txt
index 1234567..abcdefg 100644
--- a/hello.txt
+++ b/hello.txt
@@ -1 +1,2 @@
 はじめてのGit
+Git の練習中
```

| 記号           | 意味                                       |
| -------------- | ------------------------------------------ |
| `---` と `+++` | 変更前（`---`）と変更後（`+++`）のファイル |
| 赤色の `-` 行  | 削除された行                               |
| 緑色の `+` 行  | 追加された行                               |
| 記号なしの行   | 変更されていない行（文脈表示）             |

VS Code では、「ソース管理」パネルで変更ファイルをクリックすると、同じ情報がグラフィカルに表示される。左右に分割された画面で、赤（削除）と緑（追加）がハイライトされるので、CLI よりも直感的である。

### git log — 記録の履歴を確認する

`git log` にはいくつかの便利なオプションがある。

```bash
# コンパクトに一覧表示（最も使う）
git log --oneline

# 変更されたファイル名も表示
git log --oneline --name-only

# グラフ表示（ブランチの分岐が視覚的にわかる、第6章以降で便利）
git log --oneline --graph
```
