## ここから先に学ぶべきこと

Git と AI コーディングの世界は広い。本書で扱わなかったテーマの中から、次に学ぶ価値が高いものを厳選して紹介する。

### 1. VS Code 拡張機能で Git をもっと便利に

VS Code には、Git 操作を強化する拡張機能が豊富にある。

| 拡張機能        | 何ができるか                                                                         |
| --------------- | ------------------------------------------------------------------------------------ |
| **Git Graph**   | コミット履歴をグラフ（路線図のような図）で視覚化。ブランチの分岐・合流が一目でわかる |
| **GitLens**     | コードの各行に「誰がいつ変更したか」を表示。AI が変更した箇所の特定に便利            |
| **Git History** | ファイルの変更履歴を時系列で確認。過去のバージョンとの比較が容易                     |

いずれも VS Code の拡張機能パネル（左側の四角アイコン）から検索・インストールできる。

### 2. ブランチ戦略 — チーム開発の型を知る

本書ではブランチの基本だけを扱ったが、チームで開発する場合は「ブランチの運用ルール」がある。

#### GitHub Flow（最もシンプル、おすすめ）

![GitHub Flow](/images/book-10-github-flow.png)

ルールは 2 つだけ。

1. main ブランチは常に動く状態を保つ
2. 新しい作業は必ずブランチを切ってから始め、PR 経由でマージする

#### Git Flow（大規模プロジェクト向け）

main / develop / feature / release / hotfix の 5 種類のブランチを使う。ルールが複雑なので、小規模なら GitHub Flow で十分。

### 3. .gitignore のテンプレート

プロジェクトの種類ごとに、「無視すべきファイル」のテンプレートが公開されている。

GitHub でリポジトリを作成するとき、「Add .gitignore」のドロップダウンから選ぶだけで自動的に追加される。よく使うテンプレート:

- **Node** — `node_modules/`, `dist/`, `.env` 等
- **Python** — `__pycache__/`, `*.pyc`, `.venv/` 等
- **macOS** — `.DS_Store`, `._*` 等

### 4. SSH 接続 — パスワード入力を省略する

本書では HTTPS 接続（URL が `https://` で始まる方式）を使ったが、 **SSH 接続** に切り替えると、push / pull のたびにパスワード入力を求められなくなる。

設定は少し手間がかかるが、一度設定すれば以降ずっと快適になる。

大まかな手順:

1. SSH キーペア（鍵のペア）を生成する
2. 公開鍵を GitHub に登録する
3. リモート URL を SSH 形式（`github.com:ユーザー名/リポジトリ名.git`）に変更する

### 5. AI コーディングの応用

Claude Code の基本は第 8 章で学んだ。Google Antigravity も同様にターミナルから日本語で指示を出して AI にコードを書いてもらうツールであり、以下のヒントは共通して活用できる。

| テクニック              | 内容                                                                                          |
| ----------------------- | --------------------------------------------------------------------------------------------- |
| **CLAUDE.md / AGENTS.md** | プロジェクトのルートに指示ファイルを置くと、AI がプロジェクトの文脈を理解してくれる。Claude Code は `CLAUDE.md`、Google Antigravity は `AGENTS.md` が同等の役割を持つ |
| **Issue 駆動開発**      | GitHub Issue を書いてから AI ツールに「Issue #X を解決して」と指示する流れ                    |
| **PR レビューの習慣化** | AI が作った PR は必ず「Files changed」を確認してからマージする                                |
| **小さく頼む**          | 「アプリ全体を作って」より「ログインページだけ作って」のほうが精度が高い                      |

## 本書で学んだコマンド一覧

最後に、本書で登場した Git コマンドを一覧にまとめる。AI コーディング中に「このコマンド、何だっけ？」と思ったら、ここに戻ってきてほしい。

### 基本操作

| コマンド                     | 意味                   | 学んだ章 |
| ---------------------------- | ---------------------- | -------- |
| `git init`                   | リポジトリを作成       | 第 3 章  |
| `git status`                 | 現在の状態を確認       | 第 3 章  |
| `git add ファイル名`         | ファイルをステージング | 第 4 章  |
| `git add .`                  | 全変更をステージング   | 第 4 章  |
| `git commit -m "メッセージ"` | 変更を記録             | 第 4 章  |
| `git log --oneline`          | コミット履歴を表示     | 第 4 章  |

### リモート操作

| コマンド                    | 意味               | 学んだ章 |
| --------------------------- | ------------------ | -------- |
| `git remote add origin URL` | 送り先を登録       | 第 5 章  |
| `git push -u origin main`   | 初回 push          | 第 5 章  |
| `git push`                  | GitHub に送信      | 第 5 章  |
| `git pull`                  | GitHub から取得    | 第 5 章  |
| `git clone URL`             | リポジトリをコピー | 第 5 章  |

### ブランチ操作

| コマンド                   | 意味                | 学んだ章 |
| -------------------------- | ------------------- | -------- |
| `git switch -c ブランチ名` | ブランチ作成 + 切替 | 第 6 章  |
| `git switch ブランチ名`    | ブランチ切替        | 第 6 章  |
| `git merge ブランチ名`     | ブランチを統合      | 第 6 章  |
| `git stash`                | 変更を一時退避      | 第 6 章  |
| `git stash pop`            | 退避を復元          | 第 6 章  |

### 復旧操作

| コマンド                          | 意味                                   | 学んだ章 |
| --------------------------------- | -------------------------------------- | -------- |
| `git restore ファイル名`          | コミット前の変更を戻す                 | 第 9 章  |
| `git restore --staged ファイル名` | ステージングを取り消す                 | 第 4 章  |
| `git reset --soft HEAD~1`         | 直前のコミットを取り消す（変更は残る） | 第 9 章  |
| `git revert HEAD`                 | 直前のコミットを打ち消す（安全）       | 第 8 章  |
| `git reflog`                      | 操作履歴を表示（最後の砦）             | 第 9 章  |

## おわりに

Git は、最初は「なぜこんなに面倒なことを」と感じるかもしれない。しかし、AI コーディングツールを日常的に使うようになると、Git の価値が実感できるはずだ。

- AI が書いたコードを「なかったこと」にできる安心感
- 「いつ、何が変わったか」が全部記録されている安心感
- チームの誰かに「見て」と依頼できる共有の仕組み

**Git は、AI と人間が協働するための共通言語である。**

本書がその最初の一歩になれば幸いである。
