## 1. 変更を取り消す

**状況**: ファイルを変更したが、コミットする前に元に戻したい。

### VS Code での操作

1. 「ソース管理」パネルで該当ファイルを右クリック
2. 「変更を破棄」をクリック

### CLI

```bash
# 特定のファイルだけ戻す
git restore ファイル名

# すべてのファイルを戻す
git restore .
```

:::message alert
`git restore` で戻した変更は**二度と復元できない**。コミットしていない変更が対象なので、Git にも記録が残っていないからである。大事な変更がある場合は、先にコミットしてから試すこと。
:::

## 2. コミットを取り消す

**状況**: コミットしたが、やっぱり取り消したい。

### まだ push していない場合

```bash
# コミットを取り消すが、変更はファイルに残す（一番安全）
git reset --soft HEAD~1
```

これは「直前のコミットを取り消すが、ファイルの変更内容はそのまま残す」コマンドである。もう一度やり直してからコミットし直せる。

### すでに push した場合

```bash
# コミットを打ち消す新しいコミットを作成（安全）
git revert HEAD
```

revert は「直前のコミットの逆の操作」を新しいコミットとして作成する。履歴が残るので安全である。

## 3. .gitignore が効かない

**状況**: `.gitignore` にファイル名を追加したのに、Git が追跡し続けている。

**原因**: `.gitignore` に追加する **前に** すでに Git が追跡していたファイルは、追加しただけでは無視されない。

### 解決方法

```bash
# Git の追跡から外す（ファイル自体は削除しない）
git rm --cached ファイル名

# フォルダごと外す場合
git rm --cached -r フォルダ名

# 変更をコミット
git add .gitignore
git commit -m "fix: .gitignore を適用"
```

### AI コーディングで特に注意するファイル

AI がプロジェクトを作成すると、以下のファイル・フォルダが生成されることが多い。`.gitignore` に含めておくのが望ましい。

```gitignore
# 依存パッケージ（サイズが非常に大きい）
node_modules/

# 環境変数（API キーなどの秘密情報）
.env
.env.local

# ビルド成果物
dist/
build/

# OS が作る不要ファイル
.DS_Store
Thumbs.db
```

## 4. コンフリクトを解消する

**状況**: マージや pull で「CONFLICT」と表示された。

**原因**: 同じファイルの同じ箇所を、別々の場所で変更していた。

### 解消手順

1. VS Code がコンフリクト箇所をハイライト表示する
2. 以下のボタンが表示される
   - **Accept Current Change** — 自分の変更を採用
   - **Accept Incoming Change** — 相手の変更を採用
   - **Accept Both Changes** — 両方を残す
3. 適切なボタンをクリック
4. ファイルを保存
5. `git add` → `git commit` で解消を確定

:::message
コンフリクトは「Git が壊れた」わけではない。「どちらを採用するか人間に判断を求めている」だけである。
:::

## 5. 機密ファイルの誤コミット

**状況**: `.env` や API キーが入ったファイルをコミットしてしまった。

**これは最も危険なトラブルである。** 特に push 済みの場合、GitHub 上にキーが公開される可能性がある。

### まだ push していない場合

```bash
# 直前のコミットからそのファイルを除外
git reset --soft HEAD~1
git restore --staged .env
echo ".env" >> .gitignore
git add .gitignore
git commit -m "fix: .env を追跡対象から除外"
```

### すでに push してしまった場合

1. **すぐに API キーを無効化する**（各サービスの管理画面でキーを再発行）
2. ファイルを `.gitignore` に追加
3. Git の追跡から外す

```bash
git rm --cached .env
echo ".env" >> .gitignore
git add .gitignore
git commit -m "fix: 機密ファイルを除外"
git push
```

:::message alert
**GitHub に一度 push された情報は、コミットを消しても履歴に残る可能性がある。** API キーやパスワードが漏洩した場合は、Git 側の対処だけでなく、**必ずキー自体を無効化・再発行すること**。
:::

## 6. push できない

**状況**: `git push` で「no upstream branch」エラーが出る。

**原因**: ローカルのブランチが GitHub のどのブランチに対応するか、まだ設定されていない。

### 解決方法

```bash
git push -u origin ブランチ名
```

例えば、main ブランチなら:

```bash
git push -u origin main
```

`-u` をつけると、次回以降は `git push` だけで同じ送り先に push される。

## 7. detached HEAD

**状況**: 「detached HEAD」と表示されて、ブランチにいない状態になった。

**原因**: 特定のコミットに直接移動した場合に発生する。AI ツールの操作で意図せず起きることもある。

### 解決方法

作業途中の変更がない場合は、単にブランチに戻れば解決する。

```bash
git switch main
```

変更を保存したい場合は、新しいブランチを作って退避する。

```bash
git switch -c 退避用ブランチ名
```

## 8. main への誤コミット

**状況**: ブランチを切り忘れて、main に直接コミットしてしまった。

### 解決方法（直前の 1 コミットの場合）

```bash
# 1. 新しいブランチを作成（コミットを引き継ぐ）
git switch -c feature/移動先ブランチ

# 2. main に戻る
git switch main

# 3. main から直前のコミットを取り消す（変更は消えない）
git reset --soft HEAD~1

# 4. main の変更を捨てる
git restore --staged .

# 5. ブランチに戻って作業を続ける
git switch feature/移動先ブランチ
```

## 9. reset --hard の復旧

**状況**: `git reset --hard` を実行してしまい、コミットと変更が消えた。

**朗報**: Git は「消した」コミットも、しばらくの間は内部に保持している。

### 解決方法

```bash
# 操作履歴を表示
git reflog
```

こんな表示が出る。

```
a1b2c3d HEAD@{0}: reset: moving to HEAD~3
f4e5d6c HEAD@{1}: commit: 大事な変更
```

戻りたいコミットのハッシュ（左側の英数字）を見つけて:

```bash
git reset --hard f4e5d6c
```

:::message
`git reflog` は Git の「操作ログ」である。コミットの削除・リセットなど、あらゆる操作の履歴が記録されている。困ったときの最後の砦として覚えておいてほしい。
:::

## 10. 改行コード問題

**状況**: `warning: LF will be replaced by CRLF` という警告が出る。

**原因**: Windows は改行コードに `CRLF`（2 文字）、Mac/Linux は `LF`（1 文字）を使う。チームで異なる OS を使っていると、差分が改行コードだけで大量に出ることがある。

### 解決方法

Git の設定で自動変換を有効にする。

```bash
# Windows の場合
git config --global core.autocrlf true

# Mac の場合
git config --global core.autocrlf input
```

プロジェクト単位で統一したい場合は、`.gitattributes` ファイルをリポジトリのルートに作成する。

```
* text=auto
*.js text eol=lf
*.css text eol=lf
*.html text eol=lf
```

## まとめ — 困ったときの 3 ステップ

1. **落ち着く** — Git で本当に取り返しのつかないことは稀
2. **この章の逆引き表を見る** — 状況に合った対処法を探す
3. **`git reflog` を確認する** — 最後の砦。大抵のことは復旧できる
