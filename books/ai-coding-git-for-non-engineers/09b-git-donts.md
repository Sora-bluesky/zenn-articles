## この章のゴール

> この章を終えると、Git 操作で「被害が大きい」「復旧が難しい」10個の禁止事項がわかり、トラブルを未然に防げるようになる。

## はじめに

Git は間違った操作をしても、ほとんどの場合は復旧できる。`git reflog` という最後の砦があるし、コミットを消したつもりでも内部にはしばらく残っている（第 9 章参照）。

ただし、以下の 10 項目は **被害が大きいか、復旧が難しい操作** である。事前に知っておくことが最大の防御になる。

---

## 1. `git push --force` を使わない

**なぜダメか**

`git push --force` は、GitHub 上の履歴を自分のローカルの履歴で **強制的に上書きする** コマンドである。

自分ひとりで作業しているときは問題が起きにくいが、チームで使っているブランチに `--force` すると、他のメンバーがすでに push した変更が **跡形もなく消える**。しかも、消えたことに誰も気づかないまま作業が進む。

AI コーディングツールが「push に失敗しました。force push しますか？」と聞いてくることがある。反射的に「はい」と答えないこと。

**やってしまったらどうするか**

上書きされた側のメンバーのローカルに元の履歴が残っていれば、そこから復旧できる。自分しか使っていないブランチなら、`git reflog` で元のコミットを探して `git reset` で戻す。

どうしても force push が必要な場面では、代わりに `git push --force-with-lease` を使う。これは「自分が最後に取得した状態から変わっていなければ上書きする」という安全装置付きのコマンドである。

---

## 2. `.env` や API キーをコミットしない

**なぜダメか**

`.env` ファイルには API キー（外部サービスへの認証情報）やパスワードが書かれていることが多い。これをコミットして GitHub に push すると、 **世界中から読める状態になる**（パブリックリポジトリの場合）。

恐ろしいのは、後からファイルを削除しても **Git の履歴に残り続ける** ことだ。悪意のある第三者は過去の履歴をたどって秘密情報を取得できる。

**やってしまったらどうするか**

1. **まず API キーを無効化する**（各サービスの管理画面でキーを再発行）
2. `.gitignore` に `.env` を追加する
3. `git rm --cached .env` で Git の追跡から外す

詳しくは第 9 章の「機密ファイルの誤コミット」を参照。

---

## 3. `node_modules/` や `dist/` をコミットしない

**なぜダメか**

`node_modules/` は、プロジェクトが使っている外部ライブラリ（部品）が大量に入ったフォルダである。数万ファイル、数百 MB になることも珍しくない。 `dist/` はビルド（変換・圧縮）した成果物が入るフォルダである。

これらをコミットすると、リポジトリのサイズが膨れ上がる。`git clone` に何分もかかるようになるし、差分の確認も困難になる。

AI にプロジェクトを作ってもらうと、これらのフォルダが自動生成される。フォルダ自体は必要だが、Git で管理する必要はない。

**やってしまったらどうするか**

`.gitignore` にフォルダ名を追加し、追跡を外す。

```bash
# .gitignore に追加
echo "node_modules/" >> .gitignore
echo "dist/" >> .gitignore

# Git の追跡から外す（フォルダ自体は消えない）
git rm --cached -r node_modules/
git rm --cached -r dist/

git add .gitignore
git commit -m "fix: 不要フォルダを .gitignore に追加"
```

AI に「.gitignore も作って」と頼めば、プロジェクトに適した内容を生成してくれる。最初に頼んでおくのが一番確実だ。

---

## 4. main ブランチで直接作業しない

**なぜダメか**

main ブランチはプロジェクトの「正式版」である（第 6 章参照）。ここで直接作業すると、AI が予想外のコードを書いたとき、正式版が壊れる。

ブランチを切っていれば、「やっぱりダメだ」と思ったらブランチごと捨てるだけで済む。main は無傷のままだ。

**やってしまったらどうするか**

まだ push していなければ、第 9 章の「main への誤コミット」の手順でブランチに移動できる。

```bash
# 新しいブランチを作成（コミットごと引き継ぐ）
git switch -c feature/移動先

# main に戻る
git switch main

# main から直前のコミットを取り消す（変更はブランチに残っている）
git reset --soft HEAD~1
git restore --staged .
```

---

## 5. `.git` フォルダを手動で編集しない

**なぜダメか**

`.git` フォルダは、Git がすべての履歴・設定・内部データを管理している場所である。第 3 章で「箱の中にある記録帳」と説明したが、もう少し正確に言えば **Git の脳** だ。

中身をテキストエディタで直接開いて編集したり、ファイルを削除したりすると、リポジトリ自体が壊れて一切の Git 操作ができなくなる可能性がある。

**やってしまったらどうするか**

軽微な変更であれば `git fsck`（整合性チェック）で修復できることがある。

```bash
git fsck --full
```

修復不可能な場合は、GitHub にリモートの正本があれば、フォルダごと削除して `git clone` で取り直すのが最も確実である。ローカルにしかないコミットは失われる。

---

## 6. 大きなバイナリファイルをそのままコミットしない

**なぜダメか**

画像（PSD、AI）、動画（MP4）、圧縮ファイル（ZIP）などのバイナリファイルは、テキストファイルと違って **差分を圧縮できない**。1 MB の画像を 10 回変更すると、Git の内部に 10 MB 分の履歴がたまる。

リポジトリが巨大化すると、clone に時間がかかり、GitHub の容量制限（1 リポジトリ推奨 1 GB）にも引っかかる。

**やってしまったらどうするか**

今後のファイルについては、 **Git LFS（Large File Storage）** という仕組みを使うと、大きなファイルを別の場所に保管しつつ Git で管理できる。

```bash
# Git LFS の導入（初回のみ）
git lfs install

# 対象ファイルを指定
git lfs track "*.psd"
git lfs track "*.mp4"
```

本書では Git LFS の詳細は扱わないが、「大きなファイルは別の仕組みがある」ということだけ覚えておいてほしい。小さなスクリーンショット（PNG、数十 KB 程度）は通常のコミットで問題ない。

---

## 7. 意味のないコミットメッセージを書かない

**なぜダメか**

「修正」「update」「fix」だけのコミットメッセージは、後から見たときに **何を変えたのか一切わからない**。

AI コーディングでは、Claude Code が自動でコミットメッセージを生成してくれることが多い。しかし AI が「minor fix」のような曖昧なメッセージを提案してきたら、承認する前にもう少し具体的にしてもらうとよい。

3 ヶ月後の自分は、完全な他人だ。その他人が読んで「ああ、あの変更か」とわかるメッセージを書く。

**やってしまったらどうするか**

まだ push していなければ、直前のコミットメッセージは書き換えられる。

```bash
git commit --amend -m "feat: ログインボタンの色を青から緑に変更"
```

:::message
`--amend` は「直前のコミットを修正する」コマンドである。ただし、push 済みのコミットには使わないこと（次の項目 8 を参照）。
:::

---

## 8. push 済みのコミットを amend しない

**なぜダメか**

`git commit --amend` は、直前のコミットを「書き換える」操作である。ローカルだけなら問題ないが、すでに push したコミットを amend すると、GitHub 上の履歴とローカルの履歴がずれる。

結果として、次に push しようとすると拒否される。ここで `--force` に手を出すと、項目 1 の問題が発生する。

**やってしまったらどうするか**

push 済みのコミットを修正したい場合は、amend ではなく **新しいコミットを追加する** のが安全な方法だ。

```bash
# 修正内容を新しいコミットとして追加
git add .
git commit -m "fix: ログインボタンの色指定を修正"
git push
```

履歴が 1 行増えるが、チームメンバーの環境を壊すよりはるかにましである。

---

## 9. 他人のブランチを勝手に reset しない

**なぜダメか**

`git reset` はコミットの巻き戻しに使うコマンドだが、他人が作業中のブランチで実行すると、 **そのメンバーの作業が丸ごと消える可能性がある**。

自分のブランチで `reset` するのは自由だが、共有ブランチ（main や develop など）や他人が作ったブランチでの `reset` は、事前に必ず声をかけること。

**やってしまったらどうするか**

影響を受けたメンバーのローカルに元のコミットが残っていれば、そこから `git push --force-with-lease` で復旧してもらう。残っていない場合は `git reflog` で探す。

いずれにせよ、謝罪と報告が先だ。黙っていると被害が拡大する。

---

## 10. エラーメッセージを読まずにコマンドを連打しない

**なぜダメか**

Git のエラーメッセージは全て英語だが、実は **毎回「何が起きて、どうすればいいか」を教えてくれている**。

例えば:

```
hint: Updates were rejected because the remote contains work that you do
hint: not have locally. Integrate the remote changes
hint: (e.g., 'git pull ...') before pushing again.
```

これは「リモートに自分が持っていない変更がある。まず `git pull` してから push してね」という親切なメッセージだ。

ここで「なぜかうまくいかない」と思って `git push` を連打したり、`--force` を付けたりすると、状況が悪化する。

**やってしまったらどうするか**

落ち着いて、エラーメッセージを読む。英語が苦手なら、メッセージをそのまま翻訳ツールに貼り付ければよい。Claude Code を使っているなら、「今のエラーメッセージの意味を教えて」と聞けば日本語で説明してくれる。

第 9c 章の「エラーメッセージ逆引き辞典」に、よく出るエラー 20 個の和訳と対処法をまとめてある。ブックマークしておくと便利だ。

## まとめ — 10 の禁止事項チェックリスト

| # | やってはいけないこと | 被害度 |
|---|-------------------|-------|
| 1 | `git push --force` | 他人の変更が消える |
| 2 | `.env` / API キーをコミット | 秘密情報が公開される |
| 3 | `node_modules/` / `dist/` をコミット | リポジトリが巨大化する |
| 4 | main で直接作業 | 正式版が壊れる |
| 5 | `.git` フォルダを手動編集 | リポジトリが壊れる |
| 6 | 大きなバイナリをそのままコミット | リポジトリが巨大化する |
| 7 | 意味のないコミットメッセージ | 後から何の変更かわからない |
| 8 | push 済みコミットを amend | 履歴がずれる |
| 9 | 他人のブランチを勝手に reset | 他人の作業が消える |
| 10 | エラーを読まずにコマンド連打 | 状況が悪化する |

共通するのは、「 **やる前に一度立ち止まる** 」という姿勢だ。Git は優秀な安全装置を備えているが、安全装置を迂回する操作（force、reset --hard、.git の直接編集）だけは、取り返しがつかないことがある。
