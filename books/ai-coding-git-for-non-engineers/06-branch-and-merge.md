## この章のゴール

> この章を終えると、ブランチ（枝分かれ）を使って安全に新しいことを試せるようになり、Claude Code が自動でブランチを作る理由がわかるようになる。

## ブランチとは何か

ブランチ（branch）は、直訳すると「枝」である。

プロジェクトの「本流」から枝分かれして、 **本流に影響を与えずに作業できる仕組み** のことだ。

日常の作業に例えるなら、こういうイメージである。

> 重要な書類の原本をコピーして、コピーの方で赤入れ修正を試す。
> 修正がうまくいったら、原本に反映する。
> ダメだったら、コピーを捨てるだけ。原本は無傷。

Git のブランチも同じだ。

- **main ブランチ** = 原本（本流、正式版）
- **新しいブランチ** = コピー（試す場所）
- **マージ** = 修正を原本に反映する操作

### ブランチの仕組みをもう少し詳しく

ブランチは実際にはファイルのコピーを作っているわけではない。 **「どのコミット（記録）を指しているか」というポインタ（目印）** である。

![ブランチの仕組み](/images/book-06-branch.png)

この図では、A → B → C と記録が積み上がった後、C の時点で枝分かれして D → E と別の記録が続いている。

- **main** ブランチは「C を指す目印」
- **feature** ブランチは「E を指す目印」

ブランチを切り替えると、Git は該当するコミット時点のファイルを作業ディレクトリに展開する。だから、main に戻るとブランチで作ったファイルが「消える」ように見えるのである（実際は Git の中に保存されている）。

### なぜ AI ツールでブランチが重要なのか

Claude Code や Google Antigravity に「新しい機能を追加して」と頼むと、多くの場合、 **AI ツールは自動でブランチを作ってから作業する**。

なぜか。AI が書いたコードが正しいとは限らないからだ。ブランチで作業していれば、Claude Code でも Antigravity でも、AI の変更が気に入らなかった場合にブランチごと捨てれば本流は無傷で済む。

![AI がブランチで作業→マージ](/images/book-06-ai-branch-merge.png)

## VS Code でブランチを操作する

### ブランチを作成する

1. VS Code の画面左下を見ると、 **「main」** と書かれたボタンがある
2. これをクリック
3. 上部に表示されるメニューから「新しいブランチの作成」を選択
4. ブランチ名を入力する（例: `feature/add-about-page`）
5. Enter キーを押す

画面左下の表示が「main」から「feature/add-about-page」に変わる。これで、新しいブランチに切り替わった。

:::message
ブランチ名の慣例:

- `feature/○○` — 新しい機能を追加する場合
- `fix/○○` — バグを修正する場合
- `test/○○` — 実験的に何かを試す場合
  :::

### ブランチ上で作業する

新しいブランチに切り替わった状態で、ファイルを変更してみよう。

1. `about.txt` という新しいファイルを作成
2. 内容を入力（例: 「このプロジェクトについて」）
3. `Ctrl + S` で保存
4. 「ソース管理」パネルで **+** → メッセージ入力 → **✓** でコミット

### ブランチを切り替える

1. 画面左下のブランチ名（「feature/add-about-page」）をクリック
2. 「main」を選択

main ブランチに戻ると、 **先ほど作った `about.txt` が消えている** のがわかるはずだ。main ブランチには、ブランチ上での変更がまだ反映されていないからである。

もう一度「feature/add-about-page」に切り替えると、`about.txt` が戻ってくる。

## マージ — ブランチの成果を本流に取り込む

ブランチでの作業がうまくいったら、その成果を main ブランチに **マージ（統合）** する。

### VS Code でマージする

1. まず、 **main ブランチに切り替える**（画面左下のブランチ名 → 「main」を選択）
2. 「ソース管理」パネルの **「...」メニュー** →「ブランチ」→「ブランチをマージ...」
3. マージしたいブランチ（「feature/add-about-page」）を選択
4. マージが完了する

main ブランチにいる状態で、`about.txt` が表示されていれば成功である。

### マージの仕組み — 何が起きているか

マージのとき、Git は「ブランチが分岐した時点」と「両方の最新コミット」を比較して、変更を自動で統合する。

![マージコミットの仕組み](/images/book-06-merge-commit.png)

- C → D → E がブランチでの作業
- C → F で、ブランチの成果（D, E）が main に取り込まれる
- F は **マージコミット** と呼ばれ、2 つの流れが合流した記録になる

## コンフリクト — 変更がぶつかった場合

ブランチと main で **同じファイルの同じ場所** を変更していると、Git はどちらの変更を採用すべきかわからない。これを **コンフリクト（競合）** と呼ぶ。

AI ツールを使っていると、自分が手動で変更した部分と AI が変更した部分がぶつかることがある。

### どういうときにコンフリクトが起きるか

```
main で:    hello.txt の1行目を「こんにちは」に変更
feature で: hello.txt の1行目を「おはよう」に変更

→ Git「1行目をどちらにすればいいかわからない！」→ コンフリクト
```

逆に、 **別のファイル** や **同じファイルでも別の行** を変更している場合は、Git が自動でうまく統合してくれる。コンフリクトが起きるのは「同じ箇所」を変更した場合だけである。

### コンフリクトが発生すると

VS Code が該当ファイルを開き、こんな表示が出る。

```
<<<<<<< HEAD
こんにちは
=======
おはよう
>>>>>>> feature/add-about-page
```

この記号の意味:

| 記号                  | 意味                                     |
| --------------------- | ---------------------------------------- |
| `<<<<<<< HEAD`        | ここから「現在のブランチ（main）」の内容 |
| `=======`             | 区切り線                                 |
| `>>>>>>> feature/...` | ここまでが「マージ元ブランチ」の内容     |

### 解消方法

VS Code はコンフリクト箇所に以下のボタンを表示してくれる。

- **Accept Current Change** — 自分（main）の変更を採用
- **Accept Incoming Change** — ブランチの変更を採用
- **Accept Both Changes** — 両方を残す
- **手動で編集** — ボタンを使わず、自分で最終的な内容を書く

いずれかをクリック（または手動編集）して解決した後、通常通り add → commit する。

### コンフリクト解消の実践手順

1. VS Code がコンフリクトファイルをハイライト表示する
2. 各コンフリクト箇所で、上記のボタンから適切なものを選ぶ
3. `<<<<<<<`、`=======`、`>>>>>>>` の行が **すべて消えている** ことを確認
4. ファイルを保存（`Ctrl + S`）
5. 「ソース管理」パネルで **+** → メッセージ → **✓** でコミット

:::message
コンフリクトは「Git が壊れた」のではなく、「Git が判断を人間に委ねている」だけである。AI コーディングでは、AI の変更内容を見て「自分の意図と合っているか」を判断する、大事なチェックポイントになる。
:::

:::message
コンフリクトは「Git が壊れた」のではなく、「Git が判断を求めている」だけである。落ち着いて対処すれば問題ない。
:::

## 「やらかした！」復旧コーナー

### 「間違ったブランチで作業してしまった（まだコミットしていない）」

まだコミットしていない変更は、 **stash（一時退避）** で救える。

1. 「ソース管理」パネルの **「...」メニュー** →「スタッシュ」→「変更をスタッシュに保存」
2. 正しいブランチに切り替える
3. **「...」メニュー** →「スタッシュ」→「最新のスタッシュを適用」

これで、変更が正しいブランチに移動する。

### 「ブランチを消したい」

マージが終わって不要になったブランチは削除して構わない。

1. main ブランチに切り替える
2. 「ソース管理」パネルの **「...」メニュー** →「ブランチ」→「ブランチを削除...」
3. 削除したいブランチを選択

## 裏で何が起きているか（CLI 解説）

| VS Code の操作      | CLI コマンド               | 意味                               |
| ------------------- | -------------------------- | ---------------------------------- |
| ブランチ作成 + 切替 | `git switch -c ブランチ名` | 新しいブランチを作って切り替え     |
| ブランチ切替        | `git switch ブランチ名`    | 既存ブランチに切り替え             |
| マージ              | `git merge ブランチ名`     | 指定ブランチを現在のブランチに統合 |
| 一時退避            | `git stash`                | 変更を一時的に退避                 |
| 退避を戻す          | `git stash pop`            | 退避した変更を復元                 |
