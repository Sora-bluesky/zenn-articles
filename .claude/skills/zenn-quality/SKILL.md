# Zenn記事品質保証（非エンジニア向け）

## Overview

非エンジニア向けのZenn記事で、AIが陥りやすい失敗パターンを防ぐための品質基準。

## Instructions

### チェック1: 選択肢は1つに絞る
複数の方法を並列で提示しない。最も簡単な方法だけ説明。

### チェック2: プロンプト例は自然な日本語
ファイルパスを含む指示は非エンジニアには思いつかない。

### チェック3: 具体例は初心者向け
「ログイン機能」「API」ではなく「タイマーアプリ」「電卓」。

### チェック4: 暗黙の前提を説明する
「なぜそうなるの？」と思われる箇所は1文で補足。

### チェック5: エンジニア用語を避ける
src/、dist/、リファクタリングなどは説明なしに使わない。

### チェック6: Zenn記法を確認
見出し前に空行、:::message は補足、:::message alert は警告。

### チェック7: Frontmatter を確認
Zenn記事には以下の Frontmatter が必須：
- title: 記事タイトル
- emoji: 絵文字1つ
- type: "tech" または "idea"
- topics: タグ（5つまで）
- published: true/false

### チェック8: topics は人気のものを使う
Zennで人気のtopicsを優先的に使う（閲覧数に影響）：
- claudecode, ai, 生成ai, llm, windows, 個人開発 など
- Zenn公式の人気topics一覧を参考に選ぶ

### チェック9: コマンド・記号は説明する
初心者が知らない要素には補足を追加：
- irm, iex などのPowerShellコマンド
- ~（チルダ）などの記号
- PATH, 環境変数などの概念

### チェック10: 料金情報を明記する
有料サービスを扱う場合、以下を必ず記載：
- 無料で使えるか、有料か
- 既存の契約（Claude Pro/Max等）で使えるか
- 従量課金の場合は目安金額
- 無料で試す方法があれば記載

### チェック11: 認証情報の説明
認証が必要なツールの場合：
- 何の認証が必要か明記
- 自動同期される場合はその旨を説明
- ログメッセージの意味を正確に説明（誤解を招かない）

### チェック12: シリーズ記事の整合性
シリーズ記事の場合：
- 全記事のシリーズ構成を統一
- 相互リンクを適切に設定
- 参照先が実際に存在することを確認

### チェック13: 画像参照の確認
画像を使用する場合：
- `/images/xxx.png` の参照先が存在するか
- 存在しない場合はテキスト説明に置き換える

### チェック14: 公開前の最終確認（必須）

記事を公開する前に以下を必ず確認：

#### Git 操作
- [ ] `git status` で未コミットファイルがないか確認
- [ ] `git add` → `git commit` が完了しているか
- [ ] `git push` が完了しているか

#### デプロイ確認
- [ ] Zenn のデプロイが完了しているか（数分待つ）
- [ ] Zenn 管理画面に記事が表示されているか
- [ ] 公開 URL で記事が閲覧できるか

:::message alert
**過去の失敗事例1: Git プッシュ忘れ**
記事ファイルを作成し `published: true` に設定したが、Git にコミット・プッシュし忘れて公開されなかった。「ファイルを作成した」と「公開された」は別のステップであることを忘れない。

**過去の失敗事例2: Zenn レート制限**
複数の新規記事を同時に公開しようとしたが、Zennの投稿数上限に引っかかりデプロイされなかった。新規記事は1日1〜2記事程度に抑えるか、時間を空けて公開する。
:::

### チェック15: 実機検証済み確認（ツール系記事）

ツールのインストール・セットアップ・API連携を扱う記事：
- 記事の手順は筆者が実際に実行して動作確認したか
- 検証環境（OS、バージョン）が明記されているか
- 検証日時が記載されているか（APIの仕様は変わりやすい）
- 環境依存の制約（WSL2必須など）が冒頭に警告されているか

### チェック16: 料金・クォータ情報（LLMツール記事）

LLMプロバイダー（Anthropic, OpenAI, Google等）を使うツールの場合：
- 無料枠の有無と制限が記載されているか
- 他サービスとのクォータ共有について注意があるか（例: AI StudioとClawdBotで共有）
- 課金有効化の手順（必要な場合）が記載されているか
- 429エラー（RESOURCE_EXHAUSTED）の対処法が記載されているか

:::message alert
**過去の失敗事例3: クォータ共有の見落とし**
Google Gemini を使用するツールで、Google AI Studio で既にクォータを消費していたため 429 エラーが発生した。同じAPIキー/プロジェクトを使う全てのアプリでクォータが共有されることを記事に明記する必要がある。
:::

## Constraints

- 選択肢は1つに絞る
- プロンプト例にファイルパスを含めない
- 具体例は初心者向け
- 暗黙の前提を省略しない
- 検証なしで「完成」と言わない
- 有料サービスは料金情報を必ず記載
- 存在しない画像を参照しない
- シリーズ記事は構成を統一する
- **記事公開時は Git プッシュとデプロイ確認を必ず行う**
