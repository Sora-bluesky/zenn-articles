# バズポスト作成・最適化スキル

## Overview

X（Twitter）の「さらに表示」折り畳みの仕組みを理解し、フォールド前に最大限のフックを配置するポスト作成・最適化を行う。

## Instructions

### Step 1: 入力の確認

ユーザーからポスト本文を受け取る。以下のいずれかのパターン：
- **新規作成**: 記事URLやテーマを受け取り、バズるポストを作成
- **最適化**: 既存のポスト文を受け取り、「さらに表示」位置を分析・最適化

### Step 2: 「さらに表示」位置の計算

#### 前提条件

- **X Premiumが必要**: 無料アカウントの投稿上限はweighted 280（全角140文字相当）。この上限内では「さらに表示」は文字数では発生しない
- **無料アカウントの場合**: PCブラウザで11行以上の投稿のみ「さらに表示」が発生する（モバイルでは不可）

#### weighted length の計算方式（twitter-text v3準拠）

Xは `twitter-text` ライブラリのweighted length方式で文字数を計算する。**表示上の文字数ではない。**

| 文字種 | weight | 例 |
|---|---|---|
| 日本語（全角: ひらがな、カタカナ、漢字、全角記号） | **2** | あ、漢、カ、。、「」、？ |
| 半角英数・半角記号・改行・スペース (U+0000-U+10FF) | **1** | A, 1, space, \n |
| 引用符 (U+2010-U+201F) | **1** | "、"、'、' |
| プライム記号 (U+2032-U+2037) | **1** | ′、″ |
| 絵文字・その他 (上記以外) | **2** | 😀、↓、→ |
| URL | **23固定** | https://t.co/xxxx |

#### 「さらに表示」の発生条件

| 条件 | 「さらに表示」 | 対象 |
|---|---|---|
| weighted length > 280 | **あり** | 全端末（Premium必須） |
| 11行以上（weighted ≤ 280） | **あり** | PCブラウザのみ |
| weighted length ≤ 280 かつ 10行以下 | **なし** | — |

#### 折り畳み位置（フォールドポイント）の決定

weighted length > 280 の場合:

```
1. テキストを改行で「ブロック」に分割
2. 先頭から順にブロックのweightedを累積
3. 累積がweighted 280を超えるブロック以降が「さらに表示」に隠れる
4. つまり「累積 ≤ 280の最後のブロック末尾」がフォールドポイント
```

#### 計算手順

```
1. 各文字のweightを上の表に基づいて判定
2. weighted length = 各文字のweight合計
3. weighted length > 280 なら「さらに表示」発生
4. 改行でブロック分割し、累積weightedでフォールド位置を特定
```

#### 日本語ポストの目安

改行なしの場合: 全角141文字以上で「さらに表示」発生
改行N個の場合: 全角 (281-N)/2 文字以上で発生（端数切り上げ）

**注意**: 半角英字（Claude, Code, AI等）はweight 1のため、英単語が多いほどweightedが小さくなる。日本語ポストでも英単語が多いと、見た目より大幅にweightedが少なくなる

#### 重要: X の文字数カウントは UTF-16 単位

X のすべての文字数制限（ポスト280 / プロフィール160）は **UTF-16 code units** で計測される。
JavaScriptの `.length` プロパティと一致する。`[...text].length`（Unicode code points）ではない。

| 文字種 | UTF-16 units | 例 |
|---|---|---|
| BMP文字（ほとんどの文字） | 1 | あ、A、→、｜ |
| サロゲートペア（絵文字等） | **2** | 👉、😀、🔧 |

**計算時は必ず `.length`（UTF-16）で確認すること。** `[...text].length` を使うと絵文字1個につき1文字分ずれる

### Step 3: プレビュー表示

以下のフォーマットで分析結果を表示する：

```
═══ さらに表示チェッカー ═══

📊 文字数情報
- 全体: XXX文字
- フォールド位置: XX文字目（Y行目末尾）

📱 表示プレビュー

┌─── ここから表示される ───┐

（フォールド前のテキスト）

└─── ▼ さらに表示 ─────────┘

（フォールド後のテキスト）

═══════════════════════════
```

### Step 4: バズ最適化チェック

以下の観点で分析・提案を行う：

#### フォールド前（最重要）
- [ ] **フック**: 最初の1〜2行で「え？」と思わせる要素があるか
- [ ] **共感・痛み**: ターゲットの悩みに刺さる表現か
- [ ] **続きを読みたくなるか**: フォールド直前で「続きが気になる」状態を作れているか
- [ ] **改行の戦略的配置**: フォールドポイントが意図した位置にあるか

#### フォールド後
- [ ] **具体的な解決策**: フック→解決の流れが自然か
- [ ] **CTA**: リンク、フォロー誘導などがあるか
- [ ] **余韻**: 最後の1行で印象を残せているか

#### 全体
- [ ] **ターゲット明確性**: 誰に向けた投稿か明確か
- [ ] **感情トリガー**: 共感・驚き・怒り・安心のいずれかを含むか
- [ ] **具体性**: 抽象的すぎず、具体的なエピソードや数字があるか

### Step 5: 改善案の提示

問題がある場合は改善案を提示する。複数バリエーションを出す場合は最大3案まで。

各案について必ずプレビュー（Step 3のフォーマット）を表示し、フォールド位置を確認できるようにする。

## 新規作成モード

記事URLやテーマから新規にバズポストを作成する場合：

1. **ターゲット設定**: 誰に刺さるポストか明確にする
2. **フック設計**: フォールド前に収まる強力な導入を3案作成
3. **本文構成**: フック→展開→CTA の流れで構成
4. **フォールド最適化**: 各案のフォールド位置を計算し、最適な改行位置を調整
5. **プレビュー表示**: 全案のプレビューを表示

## Constraints

- 文字数計算はtwitter-text v3のweighted length方式を使うこと。表示文字数ではなくweighted lengthで判定する
- 半角英字（Claude, Code, AI等）はweight 1であり、日本語（weight 2）と混在するとweightedが見た目より少なくなることを常に意識する
- URLを含む場合はweighted 23として計算すること
- 無料アカウントではweighted 280を超える投稿自体ができないため、「さらに表示」は文字数では発生しないことをユーザーに伝えること
- 投稿前に https://x-show-more.mad-member-tools.com/ での確認を推奨すること
- 改善案は最大3案まで。選択肢を増やしすぎない
- 炎上狙いの攻撃的な表現は提案しない
- 虚偽や誇張した情報を含めない
